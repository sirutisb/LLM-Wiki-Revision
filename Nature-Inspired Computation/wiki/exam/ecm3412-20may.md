#### Part (a) [7 marks]

PSO (Kennedy & Eberhart, 1995) is a population-based continuous optimisation algorithm where each individual (particle) has a **position** $x_i$ (the candidate solution) and a **velocity** $v_i$. Particles are attracted toward their personal best position (**pbest**) and the swarm's global best position (**gbest**).

See [[pso]] for full detail.

**Algorithm elements:**

1. Initialise $N$ particles with random positions $x_i$ and velocities $v_i$; set $\text{pbest}_i = x_i$; set $\text{gbest}$ = best $x_i$.
2. At each iteration, for every particle:
   - **Update velocity:**
     $$v_{ij}(t+1) = v_{ij}(t) + c_1 z_1 (p_{ij} - x_{ij}(t)) + c_2 z_2 (g_j - x_{ij}(t))$$
   - **Update position:**
     $$x_{ij}(t+1) = x_{ij}(t) + v_{ij}(t+1)$$
   - Evaluate $f(x_i)$; update pbest and gbest if improved.
3. Terminate when max iterations reached, an acceptable solution is found, no improvement for $N$ iterations, or normalised swarm radius $\approx 0$.

**Three velocity components:**

| Term | Name | Role |
|------|------|------|
| $v_{ij}(t)$ | Inertia | Momentum — carries particle in current direction |
| $c_1 z_1 (p_{ij} - x_{ij})$ | Cognitive (personal) | Pulls toward particle's own best-known position |
| $c_2 z_2 (g_j - x_{ij})$ | Social (global) | Pulls toward swarm's best-known position |

$z_1, z_2 \sim U(0,1)$ are random draws each iteration, which prevents premature convergence.

**Parameters and feasible settings:**

| Parameter | Typical value | Effect |
|-----------|--------------|--------|
| Swarm size $N$ | 20–50 | Larger = better global search, slower per iteration |
| $c_1$ (cognitive coefficient) | $\approx 2.0$ | Too low: ignores personal history; too high: slow social convergence |
| $c_2$ (social coefficient) | $\approx 2.0$ | Too low: swarm is disconnected; too high: premature convergence to gbest |
| Neighbourhood size | Variable | Smaller: more diversity; larger: faster convergence |
| Max iterations | Problem-specific | Budget for function evaluations |

**Key point:** Too-low $c_1, c_2$ gives slow convergence; too-high causes oscillation and premature convergence.

#### Part (b) [7 marks]

The canonical algorithm is **Boids** (Craig Reynolds, 1987). See [[flocking-boids]].

**What it simulates:** Coordinated collective movement of birds (or fish/insects) with no central leader. Complex emergent flocking behaviour arises from three simple local rules applied identically by every agent.

**Main algorithmic elements:**

Each boid perceives its neighbours within a fixed-radius omni-directional sensory zone. At every time step, each boid steers according to three rules applied simultaneously:

| Rule | Description | Effect |
|------|-------------|--------|
| **1. Separation** | Steer to avoid crowding local flockmates (move away if too close) | Prevents collisions |
| **2. Alignment** | Steer toward the average heading of local flockmates | Synchronises direction |
| **3. Cohesion** | Steer toward the average position of local flockmates | Keeps the group together |

**Additional elements:**
- **Homogeneous agents:** all boids are identical — no leaders, no special roles.
- **Local information only:** each boid only interacts with its nearest neighbours; no global knowledge.
- **Immediate response:** no memory of past positions.
- **Emergent behaviours:** spontaneous polarisation, synchronised direction changes, flash expansion when startled, flock merging and splitting — none of these are explicitly programmed.

**Canonical name:** Reynolds' **Boids** model (1987). Originally created for realistic 3D computer animation, not optimisation.

*Note: PSO (part (a)) is the optimisation algorithm derived from the same flocking insight — if food source = optimal solution, the flock converges on it via the same three-component velocity update.*

#### Part (c)(i) [2 marks] — Perceptron vs MLP

See [[neural-networks]].

The **perceptron** (McCulloch & Pitts, 1943) is a **single-layer** neural network with no hidden layers: inputs connect directly to a single output unit via weighted connections.

- **Architecture difference:** The MLP adds one or more **hidden layers** between input and output; the perceptron has none.
- **Capability difference:** A perceptron can only learn **linearly separable** problems (it cannot solve XOR). An MLP with a hidden layer can represent any continuous function (universal approximator).
- **Data suited to:** The perceptron suits problems where the classes are linearly separable (e.g. simple binary classification). MLPs handle more complex non-linear mappings.

---

#### Part (c)(ii) [2 marks] — Self-Organising Map vs MLP

See [[self-organising-maps]].

- **Architecture difference:** A SOM is a **2D grid of neurons**, each with a weight vector of the same dimension as the input. There is no output layer producing a label; the output is the position of the Best Matching Unit (BMU) on the grid. An MLP is a layered feedforward network with input → hidden(s) → output producing a class label or value.
- **Learning:** SOM uses **unsupervised competitive learning** (no labelled targets, no backpropagation). MLPs use supervised backpropagation with labelled training data.
- **Data suited to:** SOMs suit **unlabelled high-dimensional data** where the goal is visualisation, clustering, or topology discovery. MLPs suit supervised classification or regression tasks with labelled examples.

---

#### Part (c)(iii) [2 marks] — Recurrent Neural Network vs MLP

- **Architecture difference:** A **Recurrent Neural Network (RNN)** has **feedback connections** — outputs (or hidden states) are fed back as inputs to the same or earlier layers, giving the network a form of memory over time. An MLP is strictly feedforward: information flows in one direction only (input → hidden → output), with no cycles.
- **Data suited to:** RNNs are best suited to **sequential or time-series data** (e.g. text, speech, financial time series) where each output depends on previous inputs. MLPs treat each input independently and have no memory of previous inputs.

#### Part (d)(i) [3 marks] — Fitness Landscape

See [[fitness-landscapes]].

A **fitness landscape** is a geometric metaphor that maps the search space of candidate solutions onto a surface where the height at each point represents the fitness (quality) of that solution.

More formally: given a search space $S$, fitness function $f(s)$, and mutation operator $M$, the **neighbourhood** of solution $s$ is the set of all solutions reachable by a single application of $M$. Solutions that are one mutation apart are "adjacent" — this defines the topology of the landscape.

**Key properties:**
- The shape of the landscape is determined jointly by the problem (fitness function) and the chosen encoding/mutation operator.
- **Smooth/unimodal landscapes:** one global peak; easy to optimise.
- **Rugged/multimodal landscapes:** many local optima separated by valleys; hard to optimise.
- **Deceptive landscapes:** local optima mislead the search away from the global optimum.
- **Plateaus:** large flat regions with no gradient to follow.

#### Part (d)(ii) [3 marks] — Why Hillclimbing Fails on Complex Landscapes

A **hillclimber** works by repeatedly mutating the current solution and accepting the mutant only if it is at least as good as the current solution. It never accepts a worse solution (no downhill moves).

**Reasons it fails on complex landscapes:**

1. **Gets trapped in local optima:** In a multimodal landscape, the hillclimber climbs the nearest peak. Once at a local optimum, every neighbour is worse, so it stops — even if there are much better peaks elsewhere. It has no mechanism to escape.

2. **Single-solution search:** The hillclimber maintains only one solution at a time. It explores only one region of the landscape and cannot simultaneously probe other promising regions.

3. **No crossover:** It cannot combine partial solutions from different good regions of the landscape (unlike population-based methods where crossover jumps between two known-good positions).

Population-based algorithms (EAs, PSO, ACO) address this by exploring multiple landscape regions in parallel, and using stochastic selection or diversity mechanisms to maintain coverage of the landscape.

#### Part (e) [6 marks] — Emergence in Conway's Game of Life

See [[cellular-automata]].

**Emergence** is the phenomenon whereby complex, structured, high-level behaviour arises from the repeated application of simple, local, low-level rules — the whole is greater than the sum of its parts.

**Conway's Game of Life (1970)** is a 2D cellular automaton: an infinite grid of binary cells (alive/dead) that update simultaneously each time step according to three simple rules based only on each cell's Moore neighbourhood (the 8 surrounding cells):

| Rule | Description |
|------|-------------|
| **Birth** | A dead cell with exactly 3 live neighbours becomes alive |
| **Survival** | A live cell with 2 or 3 live neighbours stays alive |
| **Death (underpopulation)** | A live cell with fewer than 2 live neighbours dies |
| **Death (overpopulation)** | A live cell with more than 3 live neighbours dies |

These rules make no reference to global patterns. Each cell only "knows" its 8 neighbours. Yet from these 4 rules, enormously complex emergent structures arise:

- **Gliders:** patterns that translate across the grid indefinitely
- **Oscillators:** patterns that cycle through a set of states repeatedly
- **Still lifes:** stable patterns that do not change
- **Spaceships:** moving patterns more complex than gliders
- **Universal computation:** the Game of Life is Turing-complete — it can simulate any computation

**Why this is emergence:** No individual cell rule says "create a glider" or "compute". Gliders, oscillators, and other structures are not programmed; they arise spontaneously from initial conditions and the interaction of cells following only local rules. The structures are a property of the collective system, not of any individual cell.

This is the core A-life insight shared by Boids (3 rules → flocking), ACO (simple ants → shortest-path finding), and evolutionary algorithms (selection + variation → complex adapted solutions).

#### Part (f) [3 marks] — Fractional Dimension Between 2 and 3

An example is the **Menger sponge** (or equivalently the **Sierpinski tetrahedron**). A more commonly cited 2D→3D example is the **surface of a highly crumpled or fractal 3D object**.

A clearer example suited to this question is the **Sierpinski pyramid (Sierpinski tetrahedron)**:

- It is constructed in 3D space but its recursive self-similar structure means it is "more than 2-dimensional" (has surface-like structure extending in all directions) but "less than 3-dimensional" (encloses no volume — it is infinitely porous).
- Its fractal (Hausdorff) dimension is $\log_2 4 / \log_2 2 = 2$... actually $\log 4 / \log 2 = 2$.

A better example for "between 2 and 3": the **Menger sponge**.
- Start with a $3\times3\times3$ cube; remove the central cube and the centre of each face (7 cubes removed, leaving 20).
- Repeat this process recursively on each remaining sub-cube.
- At each step the volume decreases but the surface area increases without bound.
- Fractal dimension: $d = \log(20) / \log(3) \approx 2.727$

This is strictly between 2 and 3 because the structure occupies more than a 2D surface (it extends through 3D space in a connected, space-filling way) but less than a 3D solid (it has zero volume in the limit).

**Why:** A standard 3D solid scales by $s^3$ when the linear dimension is multiplied by $s$. A standard 2D surface scales by $s^2$. The Menger sponge scales by a factor of 20 when its linear size triples ($s=3$), giving $20 = 3^d$, so $d = \log 20 / \log 3 \approx 2.727$. Since $2 < 2.727 < 3$, the dimension is indeed between 2 and 3.

#### Part (g) [5 marks] — Koch Curve: Steps and Length

**Steps to build the Koch curve:**

The Koch curve is constructed iteratively from a straight line segment:

1. **Start (iteration 0):** Begin with a single straight line segment of length 1.
2. **Divide:** Divide the segment into **three equal parts** (each of length $1/3$).
3. **Remove the middle third** and replace it with **two sides of an equilateral triangle** (a "bump") pointing outward.
4. Each straight line segment is now replaced by 4 segments each of length $1/3$.
5. **Repeat:** Apply the same rule to **every straight segment** in the current figure.
6. The Koch curve is the limit of this process as the number of iterations approaches infinity.

**Summary at each iteration $n$:**
- Number of segments: $4^n$
- Length of each segment: $(1/3)^n$
- Total length: $(4/3)^n$

**Length of the Koch curve:**

$$L_n = \left(\frac{4}{3}\right)^n$$

As $n \to \infty$:

$$L = \lim_{n \to \infty} \left(\frac{4}{3}\right)^n = \infty$$

**The Koch curve has infinite length.** At each iteration the total length is multiplied by $4/3 > 1$, so the length grows without bound. Yet the curve remains contained within a finite area — it is a fractal with dimension $d = \log 4 / \log 3 \approx 1.26$, which is between 1 (a line) and 2 (a filled area).

#### Part (a) [15 marks] — AntNet Algorithm and Experimental Findings

See [[ant-colony-optimization]], [[sources/lecture08-aco-detail]].

**Background — the routing problem:**
In a packet-switched communications network, data packets must be routed from source to destination through intermediate nodes. The goal is to minimise average packet delivery time while adapting dynamically to changing traffic loads. Traditional routing algorithms (e.g. Bellman-Ford, Dijkstra) are static and do not adapt well to network congestion.

**The AntNet algorithm (Di Caro & Dorigo, 1998):**

AntNet applies ACO principles to adaptive routing. The key idea: artificial "forward ants" and "backward ants" continuously probe the network and update local routing tables (analogous to pheromone tables) based on the quality of paths they discover.

**Main algorithm steps:**

1. **Initialisation:** Each node $n$ maintains a **routing table** $T_n$ with entries $T_n[d][k]$ = probability of forwarding a packet destined for destination $d$ via neighbour $k$. Initially set uniformly.

2. **Forward ant launch:** Periodically, each node launches a forward ant destined for a randomly chosen destination. The forward ant traverses the network hop-by-hop, choosing the next hop probabilistically based on the routing table entries (analogous to the ACO transition rule — exploiting current knowledge with some randomness).

3. **Forward ant travel:** The forward ant records the path taken and measures local statistics along the way (queueing delay, travel time). It avoids revisiting nodes (maintains a tour memory, as in standard ACO).

4. **Backward ant creation:** When the forward ant reaches its destination, it creates a **backward ant** that travels back along the same path in reverse.

5. **Pheromone update (routing table update):** As the backward ant traverses each node on the return path, it updates the routing table at that node. The update reinforces the path with a weight proportional to how good the path was (shorter travel time / lower congestion → stronger reinforcement). This is analogous to ACO's pheromone deposition rule:
   - Better paths receive stronger reinforcement
   - A forgetting (evaporation) mechanism ensures old, stale information decays

6. **Packet routing:** Real data packets are routed using the current routing tables — they take the path with the highest probability for their destination at each node.

**Structural parallels to standard ACO:**

| Standard ACO | AntNet |
|-------------|--------|
| Pheromone matrix $\tau_{ij}$ | Routing table $T_n[d][k]$ |
| Ants constructing tours | Forward ants probing paths |
| Pheromone deposition | Backward ant updates routing table |
| Pheromone evaporation | Forgetting factor in routing table |
| Heuristic $\eta_{ij}$ | Queue length at next hop |
| Construction graph | Network topology graph |

**Experimental findings (Di Caro & Dorigo, 1998):**

The paper compared AntNet against:
- **Bellman-Ford** (shortest path, static)
- **Q-routing** (a reinforcement learning approach)
- **OSPF** (Open Shortest Path First, a real-world protocol)

Key findings:
- AntNet achieved **lower average packet delivery time** than Bellman-Ford and Q-routing under varying and dynamic traffic loads.
- AntNet was **adaptive**: when traffic patterns changed (load shifted to different parts of the network), AntNet's routing tables updated to reflect new shortest/least-congested paths, whereas static algorithms continued using outdated routes.
- AntNet showed good **load balancing**: because routing decisions are probabilistic, traffic was naturally distributed across multiple paths, avoiding bottlenecks.
- Performance was robust across different network topologies and traffic intensities.
- The approach demonstrated that SI-inspired decentralised algorithms can match or outperform centralised optimisation methods in dynamic environments.

#### Part (b)(i) [5 marks] — Construction Graph

See [[ant-colony-optimization]].

A **construction graph** is the problem representation that allows ACO to be applied to problems beyond the Travelling Salesman Problem (TSP). It is a directed or undirected graph $G = (V, E)$ where:

- **Nodes** $V$ represent **partial solution components** or **decision points** — the choices available at each step of building a solution.
- **Edges** $E$ represent **allowable transitions** between components — the valid steps from one partial decision to the next.
- **Ants build solutions** by traversing a path through this graph from a start node to an end node, selecting one component at each step. The sequence of nodes visited defines a complete candidate solution.
- **Pheromone** $\tau_{ij}$ is stored on the edges (or nodes) and indicates how desirable it has been historically to move from component $i$ to component $j$.
- **Heuristic information** $\eta_{ij}$ on each edge encodes domain knowledge about the desirability of that transition (e.g. shorter distances are more desirable).

The power of the construction graph framework is that **any problem whose solution can be constructed step-by-step** can be formulated for ACO — not just TSP. Examples:
- **TSP:** nodes = cities; edges = possible next cities; one path = one complete tour.
- **Job scheduling:** nodes = jobs in sequence; one path = a complete job ordering.
- **Feature selection:** at each step: include or exclude a feature.
- **Bin packing:** layers in the graph correspond to successive item assignment decisions.

The key design choice when applying ACO to a new problem is defining the construction graph — how to decompose the solution into sequential decisions.

#### Part (b)(ii) [5 marks] — Construction Graph: 5 Variables, 3 Choices Each

A construction graph for a combinatorial problem with **5 variables** and **3 choices per variable** is a layered directed graph:

```
        [START]
       /   |   \
     v1a  v1b  v1c        ← Variable 1 (3 choices)
    / | \ / | \ / | \
  v2a v2b v2c  ...        ← Variable 2 (3 choices)
   ...
  v3a v3b v3c             ← Variable 3 (3 choices)
   ...
  v4a v4b v4c             ← Variable 4 (3 choices)
   ...
  v5a v5b v5c             ← Variable 5 (3 choices)
       |
      [END]
```

More precisely, the graph has:
- A **START** node
- **5 layers** of nodes, one per variable
- **3 nodes per layer**, one per choice value for that variable (e.g. values $\{0, 1, 2\}$)
- **Directed edges** from every node in layer $k$ to every node in layer $k+1$ (fully connected between consecutive layers)
- An **END** node connected from all nodes in layer 5

Each path from START to END passes through exactly one node per layer, making one choice per variable — forming a complete candidate solution. There are $3^5 = 243$ possible solutions.

Pheromone $\tau_{ij}$ is stored on each edge connecting layer $k$ node $i$ to layer $k+1$ node $j$, encoding how often and how successfully that combination of variable assignments has appeared in good solutions.

### Answer

#### Part (a) [6 marks] — Main Algorithmic Elements of EAs

See [[evolutionary-algorithms]], [[selection]], [[crossover-mutation]].

All evolutionary algorithms share the following common elements:

**The generic EA loop:**
```
1. Initialise population P of N random candidate solutions; evaluate fitness f(s) for each
2. Repeat until termination criterion met:
   a. SELECT parents from P (biased toward fitter individuals)
   b. VARY: apply mutation and/or crossover to produce offspring
   c. EVALUATE fitness of offspring
   d. REPLACE: update P using offspring
3. Return best solution found
```

**Core components:**
1. **Representation (encoding):** Chromosome encodes a candidate solution (binary, integer, real, permutation, tree).
2. **Fitness function:** Evaluates the quality of each solution; drives selection.
3. **Initialisation:** Typically random population; sometimes seeded with good known solutions.
4. **Selection:** Biased sampling of parents — roulette wheel, rank-based, or tournament selection.
5. **Variation operators:** Mutation (perturbation of one individual) and crossover/recombination (combining two parents).
6. **Replacement:** How offspring enter the population — generational (replace all) or steady-state (replace weakest).
7. **Termination criterion:** Max generations, acceptable fitness reached, no improvement for $k$ generations.

**Main parameters and feasible settings:**

| Parameter | Typical range | Effect |
|-----------|--------------|--------|
| Population size | 50–500 | Larger: more diversity, slower per generation |
| Mutation rate | 0.001–0.1 | Higher: more exploration; too high destroys good solutions |
| Crossover rate | 0.6–0.95 | Higher: more recombination; too low → mutation-only search |
| Tournament size | 2–10 | Larger: higher selection pressure, faster convergence, less diversity |
| Generations | 100–10,000 | More generations: larger evaluation budget |

#### Part (b) [6 marks] — GA vs GP: Similarities and Differences

See [[genetic-algorithms]], [[genetic-programming]].

**Similarities:**

- Both are EAs — they use the same generic loop: initialise → select → vary → evaluate → replace.
- Both use selection (e.g. tournament or rank-based).
- Both use mutation and crossover (recombination) as variation operators.
- Both are population-based and stochastic.
- Both are general-purpose: applicable to a wide range of problems.

**Differences:**

| Dimension | Genetic Algorithm (GA) | Genetic Programming (GP) |
|-----------|------------------------|--------------------------|
| **Individual represents** | A set of parameter values (a solution) | A program or function |
| **Chromosome structure** | Fixed-length vector (binary, integer, real, permutation) | Variable-length tree |
| **What is evolved** | Direct solution parameters | The algorithm/function that produces the solution |
| **Crossover** | Point-based (1-point, 2-point, uniform) | Subtree swapping between trees |
| **Mutation** | Gene replacement / bit-flip / Gaussian | Subtree replacement with randomly generated subtree |
| **Fitness evaluation** | Compute $f(\text{chromosome})$ directly | **Execute** the program on test cases and measure error — much more expensive |
| **Output** | A solution vector | A program (e.g. a formula, a decision tree, a robot controller) |
| **Bloat** | Not an issue (fixed length) | Programs can grow very large if depth not controlled |

**Example distinction:** A GA solving symbolic regression would encode the coefficients of a fixed polynomial $ax^2 + bx + c$ and evolve $a, b, c$. GP would evolve the structure of the formula itself — discovering that it should be quadratic in the first place.

#### Part (c) [6 marks] — Direct vs Indirect Representation

See [[representations]], [[sources/lecture05-encodings]].

**Direct representation:** Each gene directly encodes a **problem variable** — the chromosome contains the actual solution values.

**Indirect representation:** Each gene encodes a **parameter to a constructive heuristic** — a rule used to build the solution, rather than the solution itself. The decoder applies the heuristic to generate the actual solution.

**Timetabling example:**

*Problem:* Schedule 10 exams into time slots, where some exams clash (students sitting both cannot be in the same slot).

**Direct encoding:**
- Chromosome: `[4, 2, 1, 4, 3, 1, 2, 5, 3, 2]`
- Gene $i$ = the time slot assigned to exam $i$.
- Mutation: change one gene to a random slot number.
- **Problem:** Many chromosomes will be invalid (e.g. clashing exams placed in the same slot). The EA must either penalise these in the fitness function or use a repair operator. The search space includes large invalid regions.

**Indirect encoding:**
- Chromosome: `[3, 1, 2, 4, 1, 3, 2, 1, 4, 2]`
- Gene $i$ = "use the $k$th **clash-free** slot for exam $i$", where the decoder finds the $k$th slot that does not conflict with already-scheduled exams.
- Mutation: change the index for one exam to a different integer.
- **Benefit:** The decoder guarantees a clash-free (valid) timetable for any chromosome. Invalid solutions are structurally eliminated.

**Comparison:**

| Aspect | Direct | Indirect |
|--------|--------|---------|
| Invalid solutions | Common; handled by fitness penalty | Rare; handled by decoder |
| Search space | Larger (includes invalid regions) | Smaller (all chromosomes decode to valid solutions) |
| Domain knowledge required | Minimal | Yes (must design the constructive heuristic) |
| Landscape | Generally smoother | Can be more rugged |
| Decoding speed | Fast | Slower |

#### Part (d)(i) [6 marks] — Pareto Front

See [[multi-objective-optimization]], [[nsga-ii]].

In multi-objective optimisation, multiple conflicting objectives must be optimised simultaneously. Since improving one objective typically worsens another, there is no single "best" solution — instead there is a set of **trade-off solutions**.

**Pareto dominance:** Solution $\mathbf{a}$ **dominates** solution $\mathbf{b}$ ($\mathbf{a} \succ \mathbf{b}$) if:
- $\mathbf{a}$ is **at least as good as** $\mathbf{b}$ on **all** objectives, AND
- $\mathbf{a}$ is **strictly better than** $\mathbf{b}$ on **at least one** objective.

**Pareto front:** The **Pareto front** is the set of all **non-dominated** solutions — solutions for which no other solution is better on all objectives simultaneously. A solution on the Pareto front is "optimal" in the sense that you cannot improve any one objective without worsening at least one other.

**Example:** Consider minimising cost and minimising delivery time for a logistics network:

| Solution | Cost (£) | Time (hours) | Dominated? |
|----------|----------|--------------|------------|
| A | 100 | 2 | No (Pareto front) |
| B | 200 | 1 | No (Pareto front) |
| C | 150 | 3 | Yes — dominated by A (A is cheaper AND faster) |
| D | 250 | 2 | Yes — dominated by A (A is cheaper, equally fast) |

Solutions A and B are non-dominated (A is cheaper; B is faster — neither dominates the other). The Pareto front = {A, B}.

In objective space, the Pareto front is a curve (2 objectives) or surface (3 objectives) representing all optimal trade-off points.

#### Part (d)(ii) [6 marks] — Desirable Properties of the Pareto Front and Decision Making

See [[multi-objective-optimization]], [[nsga-ii]].

A good Pareto front approximation (produced by a multi-objective EA such as NSGA-II) should have two desirable properties:

**1. Convergence (closeness to the true Pareto front):**
The solutions found should be as close as possible to the true (mathematical) Pareto front — not just locally non-dominated but globally optimal trade-offs.
- *Why it matters for decision making:* If the approximated front is far from the true front, the decision-maker is choosing from sub-optimal options. They may select a solution believing it is the best possible trade-off when better ones exist.

**2. Diversity (spread and even spacing):**
The solutions on the approximated Pareto front should be:
- **Evenly spread** across the entire front — not clustered in one region.
- **Covering the full range** of trade-offs — from one extreme (optimise only objective 1) to the other (optimise only objective 2).

*Why it matters for decision making:*
- A diverse Pareto front **reveals the full trade-off landscape** to the decision-maker. They can see the cost of improving one objective at the expense of another across the full range of options.
- Clustered solutions give a **distorted picture** — many similar solutions in one region and no solutions in others, giving a false impression of what trade-offs are available.
- Evenly spaced solutions let the decision-maker **choose the trade-off that matches their priorities** (e.g. a tight budget vs. a tight deadline), rather than being forced to choose from only a narrow slice of the trade-off space.
- In practice, the decision-maker often has constraints or preferences that narrow down which region of the Pareto front is relevant — but they need a well-distributed front to make an informed choice.

**Additional benefit of multi-objective EAs:** Because EAs maintain a population, NSGA-II can approximate the entire Pareto front in **a single run**, whereas a single-objective weighted-sum approach requires $k$ separate runs to obtain $k$ Pareto front points, and misses solutions in non-convex regions of the front entirely.
