The CAP theorem
Distributed systems? Choose two:
• Consistency
Every read receives the most recent write or an error.
• Availability
Every request receives a (non-error) response.
• Partition tolerance
The system continues to operate even if parts of the network become disconnected

--- page 4 ---
The CAP theorem
• More precisely:
If there is a network partition, you have to choose between availability and consistency

--- page 5 ---
A.C.I.D. properties
• Atomicity
Each statement in a transaction (to read, write, update or delete data) is treated as a single unit. 
• Consistency
Ensures that transactions only make changes to tables in predefined, predictable ways
• Isolation
When multiple users are reading and writing from the same table all at once, isolation of their transactions
ensures that the concurrent transactions don't interfere with or affect one another
• Durability
•
Ensures that changes to your data made by successfully executed transactions will be saved, even
in the event of system failure.

Eventual consistency

--- page 7 ---
Eventual consistency
• At a given point in time, some data nodes 
in a distributed system will have an 
outdated version of the data
Write queries will arrive at different nodes at 
different times
• Most replicated data systems will provide 
an eventual consistency
All nodes will have the same data
Read requests will eventually return the same 
value

--- page 8 ---
Eventual consistency
• Can be seen as a convergence
All the nodes will converge towards the same state
• How long will it take?
Until then, reads could return anything
• These transient inconsistencies are hard to detect
They tend to appear only in case of system faults or high volume of concurrent operations

--- page 16 ---
Linearizability and the CAP theorem
• If your application requires linearizability, and some replicas are disconnected:
These replicas cannot process requests while they are disconnected (sacrificing the availability)
• Applications that don’t require linearizability can be more tolerant of network problems
• Linearizability has performance impacts
• Linearizability is not guaranteed by many distributed database systems in favour of 
performance

--- page 17 ---
Linearizability and network delays
• If you want linearizability, the response time of read and write requests is at least 
proportional to the uncertainty of delays in the network
•  In a network with highly variable delays the response time of linearizable reads and writes is 
inevitably going to be high

• Atomicity can be easily implemented on single-node databases
• What if multiple nodes are involved in a transaction (e.g., a partitioned database)?
 It is not sufficient to simply send a commit to all of the nodes and independently commit the 
transaction on each one
•
 Commits could succeed on some nodes and fail on other nodes
•
Some nodes may detect a constraint violation or conflict
•
Some of the commit requests might be lost in the network
•
Some nodes may crash

--- page 20 ---
Consensus
• Two-phase commit (2PC)
An algorithm for achieving atomic transaction commit across 
multiple nodes
The commit/abort process in 2PC is split into two phases using 
a new component called coordinator or transaction manager
•
When the application is ready to commit it sends a prepare 
request to each of the nodes
•
If all participants reply “yes,” the coordinator sends out a 
commit request
•
If any of the participants replies “no,” the coordinator 
sends an abort request to all nodes
