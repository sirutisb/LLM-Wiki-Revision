--- page 3 ---
Data models
• When we build applications, we oftentimes want to represent real-world entities (e.g., people,  
transactions, conversations) in terms of objects and data structures
• The storage of these objects will require them to be represented in terms of a data model (e.g., 
tables in an RDBM, JSON or spreadsheets).
• A data model provides the mapping between an application entity (e.g., a Python object) and 
how it will be stored in a system (e.g., tables, attributes).

The document model
• Documents encapsulate and encode data in a semi-structured standard (e.g., XML, JSON)
• Documents offer a much more natural translation from objects
All information of an object stored in a single instance in the DB
• More flexible
Not required to adhere to a rigid schema
Documents within a single store can be different from one another
• Examples: MongoDB, CouchDB (Apache SF), Firestore (Google)

Schema flexibility
• In relational DBs, schema changes can be slow and require downtime
Running UPDATE on entire tables will be slow if the table is large
 
You can always set values to NULL and update when you read the data
• Schema-on-read can be useful when:
Your data is heterogenous (i.e., different object types in the same collection)
You don’t have control over the structure of the data
 
e.g., Tweets

--- page 16 ---
Locality
• A document is usually stored in a single continuous string like JSON, XML or BSON
Pro: all the data for a given object is stored together.
• If you the application often needs to access large parts of the data, it is better than storing it across 
multiple tables.
Con:  all the data for a given object is stored together.
• When you request a document, the DB will load the entire document
If your documents are large, it can be wasteful

--- page 17 ---
Locality
• Not exclusive to document DBs
e.g. Google Spanner offers locality in a relational data model
Column-family concept used in wide-column DBs (e.g., Cassandra) will manage locality in similar 
ways

Graph-like models
• Can be seen as two relational tables, one 
for edges and one for vertices
• Any vertex can be connected to any other
• Given a vertex you can efficiently find its 
incoming and outgoing edges, making it 
easy to traverse the graph
• By using different edge labels you can 
represent many different relationships
