Reliability
• The application performs the function that the user expected. 
• It can tolerate the user making mistakes or using the software in unexpected ways. 
• Its performance is good enough for the required use case, under the expected load and data

• The system prevents any unauthorized access and abuse.

Consistency

--- page 10 ---
The CAP theorem
Distributed systems? Choose two:
• Consistency
Every read receives the most recent write or an error.
• Availability
Every request receives a (non-error) response.
• Partition tolerance
The system continues to operate even if parts of the network become disconnected

--- page 11 ---
A.C.I.D. properties
• Atomicity
Each statement in a transaction (to read, write, update or delete data) is treated as a single unit. 
• Consistency
Ensures that transactions only make changes to tables in predefined, predictable ways
• Isolation
When multiple users are reading and writing from the same table all at once, isolation of their transactions
ensures that the concurrent transactions don't interfere with or affect one another
• Durability
•
Ensures that changes to your data made by successfully executed transactions will be saved, even
in the event of system failure.

--- page 12 ---
Eventual consistency
• At a given point in time, some data nodes 
in a distributed system will have an 
outdated version of the data
Write queries will arrive at different nodes at 
different times
• Most replicated data systems will provide 
an eventual consistency
All nodes will have the same data
Read requests will eventually return the 
same value

--- page 18 ---
Replication

--- page 19 ---
Replication
• Each node stores a copy of the data (i.e., 
replica)
• Can reduce latency keeping data 
geographically close to the users
• Increase availability allowing the system 
to continue working even if some of its 
parts have failed 
• Increase performance scaling out the 
number of machines that can serve read 
queries
• Your data has to be small enough to be 
stored in a single machine
• Write operations involve updating all the 
copies of the data

--- page 20 ---
Partitioning

--- page 21 ---
Partitioning
• Each node stores a subset of the data (i.e., 
partition)
• Each piece of data will belong to exactly 
one partition
• Focus on scalability
The query load is spread across nodes
• Often times in combination with 
replication

Batch processing
Batch processing systems (offline systems)

--- page 24 ---
Stream processing

--- page 25 ---
Stream processing
• A lot of data sources are unbounded because it arrives gradually over 
time: 
E.g., Social media users produced data yesterday and today, and they will 
continue to produce more data 
• Batch processors could process data of fixed durations (e.g., every 24h)
We would always be looking at the past
We could make this window smaller and smaller…
 
… until we forget time windows and analyse every event as they arrive
 
 
i.e., Stream Processing

--- page 26 ---
Event streams
• In the stream world, data records are called events.
a small, self- contained, immutable object containing details of something that happened at some point 
in time. 
• Usually contains a timestamp 
• Can be encoded in many ways,  as a text string, JSON, binary form …
• Generated by a producer
 
AKA publisher, or sender
• Processed by multiple consumers
 
AKA Subscribers or recipients

Virtualisation
• It is the creation of a software-based 
virtual machine that acts like a real 
computer with all of its components 
(hardware, peripherals, operating system 
etc).
• A virtualisation software running a on the 
physical computer (host) allowing for one 
or more virtual computers (guests) to run 
independently from one another
