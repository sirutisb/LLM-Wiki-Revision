Introduction
• Batch processing
Read a set of files as input and produce a new set of output files. 
The output is a form of derived data; 
• dataset that can be recreated by running the batch process again if necessary 
Assumes the data is bounded
the batch process knows when it has finished reading its input

--- page 7 ---
Introduction
• Stream processing
A lot of data sources are unbounded because it arrives gradually over time: 
E.g., Social media users produced data yesterday and today, and they will continue to produce 
more data 
Batch processors could process data of fixed durations (e.g., every 24h)
We would always be looking at the past
We could make this window smaller and smaller…
… until we forget time windows and analyse every event as they arrive
i.e., Stream Processing

--- page 8 ---
Event streams
• In the stream world, data records are called events.
a small, self-contained, immutable object containing details of something that happened at some point 
in time. 
• Usually contains a timestamp 
• Can be encoded in many ways,  as a text string, JSON, binary form …
• Generated by a producer
AKA publisher, or sender
• Processed by multiple consumers
AKA Subscribers or recipients

Event streams
• Files or databases should suffice to intermediate the interaction between consumers and 
producers
Querying the database or opening the file every x seconds would add overhead
• It would be better for consumers to be notified when new events appear. 
Relational database management tools are not designed for that

--- page 11 ---
Messaging Systems 
• A common approach for notifying consumers is through a messaging system: 
a producer sends a message containing the event, which is then pushed to consumers. 
• Multiple producer nodes can send messages
• Multiple consumer nodes can receive them
• There are many approaches to this publish/subscribe model

Approaches
• Message brokers (or message queue)
essentially a kind of database optimized for message streams 
• Server  application
•
Producers and consumers connecting to it as clients. 
•
producers write messages to the broker
•
consumers receive them by reading them from the broker. 
•
Can more easily tolerate clients that come and go (connect, disconnect, and crash),

--- page 16 ---
Multiple consumers
• When multiple consumers read messages in the 
same topic
Load balancing
•
Each message is delivered to one of the consumers 
Consumers can share the work of processing the 
messages 
Fan-out
•
Each message is delivered to all of the consumers 
allows several independent consumers to subscribe to 
the same broadcast of messages
