# COM3021 May 2022 Exam

## Question 1 (20 marks)

**(a)** *In the context of large-scale data systems, provide a definition of Reliability, taking into account aspects such as user experience, system performance and security. (10 marks)*
**Solution:**
A reliable system is one that continues to work correctly (performing its expected function at the desired level of performance) even in the face of adversity. For 10 marks, key points include:
1. **Expected Function:** The system performs the function the user expects.
2. **System Performance:** The system tolerates user mistakes or unexpected usage, and performance meets the required use cases under expected load and data volume.
3. **Security:** The system prevents unauthorized access and abuse.
4. **Fault Tolerance:** A reliable system is fault-tolerant; it can anticipate faults and cope with them without them becoming full system failures.
→ [[reliability]]

**(b)** *In a large-scale distributed system, it is certain that components will fail. In this context, compare software and hardware faults, and their impact to the resilience of the system. (10 marks)*
**Solution:**
- **Hardware Faults:** These are typically independent and random (e.g., hard drive crash, RAM failure, power grid failure). Because they are uncorrelated, they can often be mitigated easily via redundancy (e.g., RAID arrays, adding more nodes).
- **Software Faults:** These are systematic errors within the system (e.g., a bug triggered by a bad input, memory leak, or unhandled exception).
- **Impact on Resilience:** Software faults are often much more damaging than hardware faults. A single software bug can cause correlated failures across many nodes (e.g., taking down the whole cluster simultaneously despite redundancy). Mitigating them requires thorough testing, process isolation, measuring, and monitoring.
→ [[fault-vs-failure]], [[reliability]]

## Question 2 (20 marks)

**(a)** *Explain one advantage and one disadvantage of a document store in comparison to a traditional relational database. (10 marks)*
**Solution:**
- **Advantage:** *Schema flexibility / Data locality.* Document stores use "schema-on-read," making it easy to adapt to changing data requirements without costly schema migrations. Data locality means retrieving an entire document is fast because all related data is stored together as a single JSON/BSON string, whereas relational DBs might require multiple expensive joins.
- **Disadvantage:** *Poor support for complex joins / Updating partial documents.* Document stores typically do not natively support complex many-to-many joins. Additionally, if a document is very large, updating or reading a single small field can be inefficient since the whole document may need to be loaded or rewritten.
→ [[document-model]], [[relational-model]], [[schema-on-read]]

**(b)** *Explain two benefits and two drawbacks of replication in the context of distributed architectures. (10 marks)*
**Solution:**
- **Benefits:**
  1. **High Availability / Fault Tolerance:** If a node goes down, the system can continue serving data from another replica.
  2. **Scalability (Read Throughput) & Latency:** Replicas can be distributed geographically closer to users to reduce latency, and multiple replicas can share the read workload, increasing overall read throughput.
- **Drawbacks:**
  1. **Complexity & Consistency Management:** Keeping replicas in sync adds complexity. Ensuring strong consistency can cause performance bottlenecks (synchronous replication), while asynchronous replication can lead to replication lag and stale reads.
  2. **Storage and Write Overhead:** Storing full copies of the data multiplies storage costs. Every write must also be propagated to replicas, which consumes network bandwidth and compute resources.
→ [[replication]], [[replication-lag]], [[synchronous-vs-asynchronous-replication]]

## Question 3 (10 marks)

**Question:** *You are designing a distributed data system for a social media platform. Which data distribution architecture would deliver a more scalable, reliable and resilient system? Justify your choice describing the advantages of your approach against other possible strategies. (10 marks)*
**Solution:**
For a social media platform handling massive scale, the best approach is **combining both Partitioning and Replication**. 
- **Partitioning (Sharding):** Using hash partitioning (e.g., on User ID or Post ID) allows the system to distribute the data and write/read load evenly across many machines, solving the scalability problem that a single-machine approach would face.
- **Replication:** Each partition should be replicated (e.g., Leader-Follower setup) across different geographic data centers. This ensures high reliability and resilience; if one node fails, a follower takes over. It also provides low latency for users globally.
- **Justification against alternatives:** Using replication alone wouldn't scale for writes or total dataset size. Using partitioning alone would leave single points of failure. The combination (with eventual consistency for performance) is standard for massive AP (Available/Partition-tolerant) systems like social media.
→ [[partitioning]], [[replication]], [[hash-partitioning]]

## Question 4 (30 marks)

**(a)** *Your system will generate daily outputs from the collected results and produce aggregated statistics from it. What choices would you make regarding the design of the data system for this solution, taking into account aspects such as scalability, reliability and data processing? Explain and justify your design choices. (15 marks)*
**Solution:**
- **Data Processing:** For daily outputs and aggregations, a **Batch Processing** framework (like MapReduce or Spark) is ideal. The input data for a single day is bounded, allowing high-throughput, offline processing. 
- **Scalability:** The batch jobs should be designed to scale horizontally across commodity hardware. Utilizing a Distributed Filesystem (e.g., HDFS) ensures that data storage scales with the processing nodes.
- **Reliability:** Batch processing frameworks like MapReduce are highly fault-tolerant. If a mapper or reducer fails during the job, the framework simply restarts the task on another node. Data stored in the distributed filesystem is replicated, ensuring data durability and reliability against hardware failure.
→ [[batch-processing]], [[mapreduce]], [[distributed-filesystem]]

**(b)** *One component of your system will provide real-time data as the test data arrive. This data will be consumed by a dashboard application showing live statistics and predictions to key decision-makers. How would you implement the server-side of this system and how would this component interact with other components (e.g., the data storage)? (10 marks)*
**Solution:**
- **Implementation:** This component requires **Stream Processing** using a Message Broker (e.g., Kafka) and a stream processing framework. 
- **Interaction:** As test data arrives, it is published as an event to the Message Broker. The stream processor consumes these events in real-time, updates sliding/tumbling window aggregations, and updates the live dashboard.
- **Data Storage:** The stream processing framework can maintain state locally for speed, but should periodically flush results to a fast, scalable data store (like a NoSQL Key-Value or Document store) designed for high read-throughput to serve the dashboard without locking up the batch processing system.
→ [[stream-processing]], [[messaging-systems]], [[message-broker]]

**(c)** *Later on it is decided that this system will provide an integration with social media platforms, allowing users to post and share the daily statistics and predictions. Another member of the team suggests using MPI to implement the social media integration through message-passing. How would you evaluate this suggestion? Justify your answer. (5 marks)*
**Solution:**
- **Evaluation:** Using MPI (Message Passing Interface) for social media integration is a **terrible suggestion**.
- **Justification:** MPI is designed for tightly coupled, high-performance computing (HPC) environments running on a dedicated cluster with a shared or distributed memory architecture. Social media integration involves loosely coupled, unreliable web APIs over the internet. Instead, a standard message broker (fan-out pattern) and asynchronous webhooks or REST APIs should be used to push updates.
→ [[mpi]], [[multiple-consumers]]

## Question 5 (20 marks)

**(a)** *In the context of distributed systems, explain the CAP theorem and the trade-off it entails. (6 marks)*
**Solution:**
- **CAP Theorem:** A distributed system cannot simultaneously guarantee **Consistency** (every read receives the most recent write or an error), **Availability** (every request receives a non-error response), and **Partition Tolerance** (the system continues to operate despite network messages being dropped/delayed).
- **Trade-off:** Because network partitions (P) are inevitable in distributed systems, the system must choose between being **CP** (sacrificing availability during a partition to ensure strong consistency) or **AP** (sacrificing strong consistency, providing eventual consistency, to remain available).
→ [[cap-theorem]]

**(b)** *Describe vertical and horizontal scaling and three key differences between them, taking into account scalability, resilience and maintainability aspects. (14 marks)*
**Solution:**
- **Vertical Scaling (Scale Up):** Adding more power (CPU, RAM) to an existing single machine.
- **Horizontal Scaling (Scale Out):** Adding more independent machines to a network to share the workload.
- **Differences:**
  1. **Scalability:** Vertical scaling has a hard hardware limit and becomes exponentially expensive at the high end. Horizontal scaling offers virtually infinite scalability by simply adding more commodity servers.
  2. **Resilience:** Vertical scaling introduces a single point of failure (if the machine goes down, the system is down), offering poor resilience. Horizontal scaling is highly resilient; if one node fails, others can take over the load (especially when combined with replication).
  3. **Maintainability:** Vertical scaling is simpler to maintain initially as it requires no complex distributed systems logic. Horizontal scaling increases complexity (e.g., data partitioning, network latency, distributed consensus), making the system harder to design and operate.
→ [[vertical-vs-horizontal-scaling]]
