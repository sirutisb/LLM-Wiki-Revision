## Question 2 (20 marks)

**(a)** *Describe the differences between software and hardware faults in the context of system availability. (10 marks)*
**Solution:**
- **Hardware Faults:** Are typically random, independent events (e.g., a hard disk crashing, a power supply failing). Because they are independent, they can be easily mitigated using hardware redundancy (like RAID or adding more servers to a cluster). If one fails, the others are highly unlikely to fail at the exact same moment.
- **Software Faults:** Are systematic and correlated (e.g., a bug in the code, a memory leak, or a flaw triggered by a specific unhandled input). Because the exact same software runs across the entire cluster, a software fault can cause every single redundant node to crash simultaneously, completely destroying system availability. Mitigation requires process isolation, exhaustive testing, and monitoring.
→ [[fault-vs-failure]]

**(b)** *When it comes to making decisions related to the design of data-intensive systems, explain the trade-off between eventual consistency and linearizability, and how choices made based on this trade-off can affect the availability of said system. (10 marks)*
**Solution:**
- **Linearizability (Strong Consistency):** Guarantees that the system behaves as a single copy; once a write is successful, all subsequent reads instantly see it. To achieve this in a distributed system, nodes must coordinate (consensus) before confirming writes/reads. During a network partition, the system must refuse requests to avoid returning stale data, severely reducing **availability**.
- **Eventual Consistency:** Guarantees only that all replicas will eventually converge if there are no new writes. It provides no recency guarantee. However, it allows nodes to accept writes and serve reads entirely independently. Even during a network partition, nodes remain highly **available** to users, answering requests instantly (though possibly with stale data).
→ [[linearizability]], [[eventual-consistency]], [[cap-theorem]]

## Question 3 (30 marks)

**Question:** *You are the Data Science Lead for a social media platform called Y.*
**(a)** *Describe in detail how you would design the data storage solution of Y.*
**(i)** *What data store model would you adopt for the user and who-follows-whom data? Justify your answer. (5 marks)*
**Solution:**
I would use a **Graph database model**. The "who-follows-whom" relationship is inherently a many-to-many network of vertices (users) and directed edges (follows). Graph databases are highly optimized for graph traversals (e.g., finding the followers of the people I follow), whereas implementing this in a relational database would require expensive, highly inefficient recursive joins.
→ [[graph-model]]

**(ii)** *As a global application used by millions of users, certain decisions have to be made to ensure the scalability of the platform. Explain and justify in detail the decisions you have made regarding the distributed data architecture and consistency guarantees. (25 marks)*
**Solution:**
- **Architecture (Partitioning & Replication):** The system must use a combination of **Partitioning** and **Replication**. The immense data volume and write-load (millions of users posting) cannot fit on one machine. I would use **Hash Partitioning** (e.g., hashing the User ID) to distribute the users and their posts evenly across hundreds of nodes, avoiding hotspots. Every partition would then be **replicated** across multiple geographic data centers using a **Leader-Follower** setup. This ensures that if an entire data center goes offline, a replica elsewhere can immediately serve the data, ensuring high availability and localizing reads for global users to reduce latency.
- **Consistency Guarantees:** I would design the system as an **AP (Available and Partition-tolerant)** architecture per the CAP theorem, offering **Eventual Consistency**. For a social media platform, it is vastly more important that the system is fast, responsive, and always available than it is for every user globally to see a new post or "like" at the exact same millisecond. Linearizability is too expensive and slow for this global scale.
→ [[partitioning]], [[replication]], [[cap-theorem]], [[leader-follower-replication]]

## Question 4 (45 marks)

**(a)** *When designing a fault-tolerant, scalable system, what are the key considerations in deciding whether to use vertical or horizontal scaling? (10 marks)*
**Solution:**
- **Vertical Scaling (Scale Up):** Adding more CPU/RAM to a single machine. It is simpler to maintain because it avoids distributed systems logic. However, it has a hard hardware limit, cost scales non-linearly (supercomputers are very expensive), and crucially, it offers zero inherent fault tolerance—if the single machine dies, the system goes offline.
- **Horizontal Scaling (Scale Out):** Adding more commodity machines to a cluster. It offers virtually infinite scalability and excellent fault tolerance (if one node dies, others take over). The trade-off is drastically increased complexity: the system must now handle network partitions, data partitioning, distributed consensus, and replication.
→ [[vertical-vs-horizontal-scaling]]

**(b)** *Explain covariate shifts and describe an illustrative scenario in which such shift occurred. (10 marks)*
**Solution:**
- **Covariate Shift:** A type of data shift where the distribution of the input variables (the independent variables / covariates) changes over time, but the underlying relationship mapping inputs to outputs remains exactly the same.
- **Scenario:** A facial recognition system is trained on images of people's faces under normal conditions. Suddenly, a pandemic occurs and everyone starts wearing face masks. The input data distribution has drastically shifted (the pixels representing faces are now mostly masks), causing the model's accuracy to plummet, even though the true mapping of "features to identity" hasn't fundamentally changed.
→ [[data-shifts]]

**(c)** *In the context of a leader-follower architecture, explain consensus and what it represents in terms of performance, consistency and availability. (15 marks)*
**Solution:**
- **Consensus:** The process by which multiple distributed nodes agree on a single value or state (e.g., agreeing on the order of writes, or agreeing on which node should be the new leader after a crash).
- **Consistency:** Consensus is the foundation of strong consistency (linearizability). It ensures that all replicas agree on the exact sequence of events, preventing split-brain scenarios and data corruption.
- **Performance & Availability:** Achieving consensus requires synchronous network communication (often involving a majority quorum). This severely degrades **performance** by adding network latency to operations. Furthermore, it harms **availability**; if a majority of nodes cannot communicate due to a network partition, the consensus protocol halts, and the system cannot process writes.
→ [[consensus]], [[leader-follower-replication]], [[cap-theorem]]

**(d)** *In a context of parallel computing, describe MPI explaining how it addresses the parallel programming model. (10 marks)*
**Solution:**
- **MPI (Message Passing Interface):** A standardized API for distributed-memory parallel computing. It is the dominant model for High-Performance Computing (HPC) clusters.
- **How it addresses parallel programming:** In a distributed-memory model (MIMD), processors do not share physical RAM; each has its own private memory. MPI addresses this by providing explicit, two-sided message passing. If Processor A computes a value that Processor B needs, A must explicitly call an `MPI_Send` command, and B must explicitly call a matching `MPI_Recv` command. MPI handles the routing, buffering, and synchronization of these messages across the cluster network, allowing developers to build massive parallel applications.
→ [[mpi]], [[thread-level-parallelism]]
