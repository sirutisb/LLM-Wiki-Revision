# Lecture: Stream Processing

*Unbounded data, continuous arrival. Stream processing is the limit of batch as the window shrinks to zero. Built on event streams, message brokers, and the producer/consumer pattern.*

> **Source note:** the 2025 deck is canonical (it adds a feedback-summary section but otherwise covers the same content as 2024). The Review deck specifically calls out **event time vs processing time** — included here as a separate concept.

## Slide-by-slide notes

- **(s. 6)** **Batch processing recap**:
  - Reads input files, produces output files (derived data, recomputable).
  - Assumes data is **bounded** — the job knows when it has finished.
- **(s. 7)** **Stream processing motivation**:
  - Many data sources are **unbounded** — social-media users keep producing data.
  - Batch can run on fixed durations (every 24h, every hour) but is always looking at the past.
  - Make the window smaller and smaller, until you treat every event as it arrives — that's stream processing.
- **(s. 8)** [[event-streams|Events]]:
  - In a stream, records are called **events**.
  - Each event is small, self-contained, **immutable**, with details of something that happened at a specific time.
  - Usually contains a **timestamp**.
  - Encoded as text, JSON, binary, etc.
  - Generated by a **producer** (publisher, sender).
  - Processed by **consumers** (subscribers, recipients).
- **(s. 10)** **Why not just databases?** Polling files or DBs every few seconds is wasteful. Consumers want to be **notified** as events arrive — relational tools aren't designed for that.
- **(s. 11)** [[messaging-systems|Messaging systems]] solve this: a producer sends a message containing the event; the system pushes it to consumers. Multiple producers, multiple consumers — the **publish/subscribe** model.
- **(s. 12)** **Backpressure** problems: what if producers send faster than consumers can process?
  - Drop messages.
  - Buffer in a queue.
  - Apply **backpressure** (flow control — slow the producer).
  - What if a consumer crashes / disconnects? Are messages lost or persisted?
- **(s. 13)** **Approach 1 — direct messaging** (multicast over TCP / UDP):
  - Producer pushes directly to consumers.
  - **Brittle** — consumer might be offline; no buffering, no replay.
- **(s. 14)** **Approach 2 — [[message-broker]]** (or message queue):
  - A server-side broker sits between producers and consumers.
  - Producers write to the broker; consumers read from it.
  - Tolerates clients that connect, disconnect, and crash.
- **(s. 15)** **Brokers vs databases**:
  | Databases | Message brokers |
  |---|---|
  | Keep data until explicitly deleted | Auto-delete after successful delivery |
  | Persistent | Queues are short |
  | Advanced search | Topic-based subscription |
- **(s. 16)** [[multiple-consumers|Multiple consumers]] — two patterns when several consumers read the same topic:
  - **Load balancing** — each message goes to *one* consumer; consumers share work.
  - **Fan-out** — each message goes to *all* consumers; multiple independent subscribers see the broadcast.
- **(s. 17)** **Acknowledgements and redelivery**:
  - A consumer must explicitly ack the broker after processing.
  - If the broker doesn't get an ack, it redelivers the message to another consumer.
  - Redelivery can affect **ordering** when load-balancing is used — messages may be processed out of original order.

## Background — material from the Review deck

The Review highlights one slide that isn't in the Stream Processing deck itself but is exam-relevant:

- **[[event-time-vs-processing-time|Event time vs processing time]]** — Confusing the two leads to bad data. *Event time* is when the event happened; *processing time* is when the system saw it. Network delays, retries, and out-of-order arrivals make these diverge. A correctly-built stream pipeline operates on event time and uses **watermarks** to handle late events.

## Background — production stream tooling

The slides cover the conceptual model but not specific tools. Worth knowing:

- **Apache Kafka** — durable distributed log, the de facto message broker. Topics are partitioned, replicated, retained.
- **Apache Flink, Apache Spark Streaming, Kafka Streams** — stream processors. Compute over events with windowing, joins, state.
- **Windowing primitives** — tumbling, sliding, session windows. Aggregate over time ranges.

## Key takeaways

1. **Stream = batch with a vanishing window.** Same conceptual lineage; different latency target and unboundedness.
2. **Events are immutable, timestamped, self-contained.** That immutability is what enables replay-based fault tolerance.
3. **A message broker decouples producers from consumers.** It absorbs bursts, handles disconnections, supports many consumption patterns.
4. **Load balancing vs fan-out** — two ways multiple consumers share or duplicate a topic.
5. **Acknowledgements + redelivery** = at-least-once semantics. The cost is potential reordering.
6. **Event time matters more than processing time.** Use the timestamp the event carries, not the time you saw it — otherwise late or out-of-order events corrupt your aggregations.
