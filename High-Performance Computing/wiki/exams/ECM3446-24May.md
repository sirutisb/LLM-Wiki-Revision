## Question 1 (40 marks)

### 1(a) Why parallel programming is unavoidable (8 marks)

(See the longer answer in [ECM3446-23May.md §1(e)](ECM3446-23May.md). Same question.)

The researcher confuses **Moore's Law** with **clock-frequency scaling**:

- **Moore's Law** (transistor density doubling) is still progressing, but it now buys *more cores per chip*, not faster cores.
- **Dennard scaling** — the rule that smaller transistors run faster *and* use less power per unit area — broke down in the mid-2000s. Leakage current and the resulting power-density wall mean further frequency increases produce more heat than air-cooling can dissipate.
- Clock frequencies have been stuck around 2–4 GHz since ~2005. Single-core gains now come only from architectural improvements (wider vector units, better branch prediction, more cache).
- The only way to convert Moore's Law's transistor budget into program performance is to use the *additional cores*. A serial program leaves all but one core idle.

So Moore's Law is not a substitute for parallelism — it is the *reason* parallelism is now mandatory.

*Concepts: [Moore's Law and Dennard Scaling](../concepts/Moores_Law_and_Dennard_Scaling.md), [HPC Programming Languages](../concepts/HPC_Programming_Languages.md)*

### 1(b) The LINPACK benchmark

#### (i) What is LINPACK? (2 marks)

LINPACK / **High-Performance Linpack (HPL)** is a benchmark that solves a dense system of linear equations $Ax = b$ using LU factorisation, measuring sustained floating-point performance in flop/s. The Top500 list ranks systems by their measured HPL performance ($R_{max}$).

*Concepts: [Performance Metrics and Top500](../concepts/Performance_Metrics_and_Top500.md)*

#### (ii) Why real-world applications underperform LINPACK (6 marks)

LINPACK is engineered to maximise the fraction of $R_{peak}$ achieved. Real applications typically fall well below it because:

1. **Arithmetic intensity** — LINPACK uses dense matrix-matrix operations (BLAS Level 3, like DGEMM) with $O(N)$ FLOPs per byte moved. It is heavily compute-bound, so the CPU is never starved for data. Most real codes (sparse solvers, finite-difference stencils, vector ops) have $O(1)$ arithmetic intensity and are **memory-bandwidth-bound**, never approaching $R_{peak}$.

2. **Vectorisation and FMA** — LINPACK's inner loops use the widest available vector units (AVX-512) with fused multiply-add. Many real codes don't, especially legacy and unoptimised software.

3. **Communication overhead** — LINPACK has been heavily tuned to overlap MPI communication with computation. Real codes that perform halo exchanges, reductions, or all-to-all collectives every iteration spend a non-trivial share of their time waiting on the network.

4. **Load imbalance** — LINPACK partitions a uniform workload across processes. Real applications often have heterogeneous work (e.g. adaptive mesh refinement) that creates idle time.

5. **Cache behaviour** — LINPACK is blocked to fit operands into L1/L2 cache. Codes that don't apply cache blocking spend much more time waiting on DRAM.

6. **I/O and other overheads** — Real applications read input, checkpoint, log progress, and synchronise — none of which LINPACK does.

The HPCG benchmark was introduced precisely because it is more representative: it solves a sparse problem, hits memory bandwidth as its bottleneck, and shows performance figures 1–2 orders of magnitude lower than HPL on the same hardware.

*Concepts: [Arithmetic Intensity and Roofline Model](../concepts/Arithmetic_Intensity_and_Roofline_Model.md), [BLAS and Dense Matrices](../concepts/BLAS_and_Dense_Matrices.md), [Memory Hierarchy and Cache](../concepts/Memory_Hierarchy_and_Cache.md), [Domain Decomposition Overheads](../concepts/Domain_Decomposition_Overheads.md), [Performance Metrics and Top500](../concepts/Performance_Metrics_and_Top500.md)*

### 1(d) Matrix operations

#### (i) Sparse vs dense (2 marks)

- **Dense matrix:** most or all elements non-zero, stored as a contiguous 2-D array.
- **Sparse matrix:** the vast majority of entries are zero; specialised formats (CSR, COO, …) avoid storing the zeros and skip the corresponding multiplications.

#### (ii) CSR format (4 marks)

Three 1-D arrays for a matrix with `nrow` rows and `nnz` non-zero entries:
1. **`vals[nnz]`** — the non-zero values, in row-major order.
2. **`col_ind[nnz]`** — the column index of each value.
3. **`row_ptr[nrow + 1]`** — `row_ptr[r]` is the index where row `r` starts in `vals`/`col_ind`. Final entry is `nnz + 1` (1-indexed) as a sentinel.

Storage: $2 \cdot nnz + nrow + 1$ entries instead of $nrow \cdot ncol$. Trade-off is **indirect addressing** through `col_ind`, slower than direct dense access.

*Concepts: [Sparse Matrices and CSR](../concepts/Sparse_Matrices_and_CSR.md)*

#### (iii) Memory bytes scaling for dense matmul (2 marks)

$O(N^2)$. Each $N \times N$ matrix has $N^2$ entries, and matmul reads $A$, reads $B$, writes $C$ — all $O(N^2)$ in memory traffic. FLOPs scale as $O(N^3)$, so AI grows as $O(N)$.

*Concepts: [BLAS and Dense Matrices](../concepts/BLAS_and_Dense_Matrices.md), [Arithmetic Intensity and Roofline Model](../concepts/Arithmetic_Intensity_and_Roofline_Model.md)*

### 1(e) Four situations to use OpenMP instead of, or in combination with, MPI (8 marks)

Pick four — give about a sentence each:

1. **Single shared-memory node target.** When the target hardware is one server / workstation with shared memory, OpenMP delivers parallelism without the marshalling overhead of MPI. A 12-core server with one OpenMP program is simpler than 12 MPI ranks on it.

2. **Incremental parallelisation of an existing serial code.** OpenMP only requires `#pragma omp parallel for` and scoping clauses; the serial control flow stays intact. MPI requires re-architecting the code to handle data distribution, halo exchange, and collective communication.

3. **Hybrid MPI+OpenMP for very large clusters.** Use MPI between nodes (one rank per node or per socket) and OpenMP within each node. This reduces the total number of MPI ranks, cuts down on duplicated halo memory, and reduces the number of network messages.

4. **Fine-grained parallelism within a region that pure MPI cannot exploit.** OpenMP `tasks` and dynamic loop scheduling can balance irregular work cheaply. Doing the same dynamic balancing through MPI requires implementing a manager-worker pattern.

5. **Quick performance win on a multicore laptop / desktop.** When the goal is "make this run in 1 hour instead of 8" and the user has one machine, OpenMP is the lowest-effort option (see Q2(b) below).

6. **Memory pressure.** If each MPI rank's halo / duplicated state is too large to fit `RAM / cores_per_node` per rank, switch to OpenMP within the node so that all cores share a single copy of the static data.

*Concepts: [OpenMP vs. MPI](../comparisons/OpenMP_vs_MPI.md), [Hybrid Parallelism (MPI + OpenMP)](../concepts/Hybrid_Parallelism_MPI_OpenMP.md), [OpenMP Tasks](../concepts/OpenMP_Tasks.md)*

## Question 2 (30 marks)

### 2(a) The missing MPI call (10 marks)

Output observed:
```
Rank: 0 a after: 0 1 2
Rank: 0 b after: 0 1 2 3 4 5
Rank: 1 a after: 3 4 5
Rank: 1 b after: 0 0 0 0 0 0
```

**Reasoning:**
- On rank 0, `b` has changed from `[0,0,0,0,0,0]` to `[0,1,2,3,4,5]`.
- That is precisely the concatenation: rank 0's `a = [0,1,2]` placed at the start, followed by rank 1's `a = [3,4,5]`.
- On rank 1, `b` is unchanged.

This pattern matches **`MPI_Gather`** from `a` into `b`, with rank 0 as the root. Each process contributes 3 ints; the root receives `nprocs × 3 = 6` ints, in rank order, into `b`.

```c
MPI_Gather(a, 3, MPI_INT, b, 3, MPI_INT, 0, MPI_COMM_WORLD);
```

**Argument-by-argument:**
| Arg | Value | Meaning |
|:---|:---|:---|
| `sendbuf` | `a` | local data each rank sends |
| `sendcount` | `3` | 3 ints per rank |
| `sendtype` | `MPI_INT` | element type |
| `recvbuf` | `b` | only used on root; ignored elsewhere |
| `recvcount` | `3` | size received from *each* rank |
| `recvtype` | `MPI_INT` | type received |
| `root` | `0` | rank 0 collects |
| `comm` | `MPI_COMM_WORLD` | communicator |

It is *not* `MPI_Allgather` — only rank 0's `b` was modified. It is *not* `MPI_Scatter` — `b` already had the right shape but didn't exist on rank 1's output. It is *not* a point-to-point send — both messages arrive cleanly indexed.

*Concepts: [MPI Collective Communication](../concepts/MPI_Collective_Communication.md)*

### 2(b) Speeding up an 8-hour serial program on a 12-core node

#### (i) Minimum parallel efficiency to hit 1 hour (4 marks)

Required speed-up: $S = T_0 / T_N = 8 / 1 = 8$ on $N = 12$ cores.

Parallel efficiency:
$$E = \frac{S}{N} = \frac{8}{12} = 0.667 = 66.7\%$$

So the implementation must achieve **at least 66.7% efficiency** on 12 cores.

*Concepts: [Parallel Scaling](../concepts/Parallel_Scaling.md)*

#### (ii) OpenMP or MPI? (6 marks)

**Choose OpenMP.** Justification:

- The target hardware is a **single 12-core node** with shared memory. OpenMP runs there natively; there's no point in paying MPI's distribution overhead for a problem that fits on one node.
- The existing code is **serial with most work in `for` loops** — exactly OpenMP's sweet spot. Adding `#pragma omp parallel for` (with appropriate `reduction`/`private` clauses) typically yields a parallel version in hours.
- The deadline is "as fast as possible". MPI would require restructuring the program to explicitly distribute arrays, code halo or gather/scatter steps, and debug deadlocks — days to weeks of work for no extra speed-up over OpenMP at this scale.
- 66.7% efficiency on 12 cores is achievable with OpenMP for a loop-dominated code, especially if the loops have no carried dependencies and operate on large enough arrays.

*Concepts: [OpenMP vs. MPI](../comparisons/OpenMP_vs_MPI.md), [OpenMP](../concepts/OpenMP.md), [Parallel Loops in OpenMP](../concepts/Parallel_Loops_OpenMP.md)*

### 2(c) CSR representation of the matrix (10 marks)

$$
\begin{pmatrix}
5 & 0 & 2 & 0 \\
0 & 1 & 0 & 0 \\
2 & 0 & 0 & 0 \\
0 & 0 & 0 & 7
\end{pmatrix}
$$

Non-zeros (1-indexed):

| Row | Non-zeros |
| :--: | :--- |
| 1 | (col 1, val 5), (col 3, val 2) |
| 2 | (col 2, val 1) |
| 3 | (col 1, val 2) |
| 4 | (col 4, val 7) |

`nnz = 5`.

```
vals    = [ 5,  2,  1,  2,  7 ]
col_ind = [ 1,  3,  2,  1,  4 ]
row_ptr = [ 1,  3,  4,  5,  6 ]
```

Cumulative count of non-zeros gives the row pointers: row 1 has 2 entries, then +1, +1, +1 → 1, 3, 4, 5, 6.

*Concepts: [Sparse Matrices and CSR](../concepts/Sparse_Matrices_and_CSR.md)*

## Question 3 (30 marks)

### 3(a) 2-D PDE solver — halo exchange (8192 × 8192 domain)

#### (i) Values per halo, 16 × 16 decomposition (2 marks)

Sub-domain size: $\frac{8192}{16} \times \frac{8192}{16} = 512 \times 512$. One halo edge to one neighbour:

$$\text{Values per halo} = 512$$

*Concepts: [Domain Decomposition](../concepts/Domain_Decomposition.md)*

#### (ii) Bytes per halo (2 marks)

$$512 \times 8 = 4096\ \text{bytes}$$

#### (iii) Transmission time, L = 1 µs, B = 15 GB/s (3 marks)

$$t = L + \frac{M}{B} = 1.0 \times 10^{-6} + \frac{4096}{15 \times 10^{9}}$$
$$= 1.0 \times 10^{-6} + 2.731 \times 10^{-7}$$
$$\boxed{t \approx 1.27 \times 10^{-6}\ \text{s} = 1.27\ \mu\text{s}}$$

*Concepts: [Interconnects and Network Topologies](../concepts/Interconnects_and_Network_Topologies.md)*

#### (iv) Transmission time with 32 × 32 decomposition (3 marks)

Sub-domain: $\frac{8192}{32} \times \frac{8192}{32} = 256 \times 256$. Halo length 256, bytes $256 \times 8 = 2048$.

$$t = 1.0 \times 10^{-6} + \frac{2048}{15 \times 10^{9}} = 1.0 \times 10^{-6} + 1.365 \times 10^{-7}$$
$$\boxed{t \approx 1.14 \times 10^{-6}\ \text{s} = 1.14\ \mu\text{s}}$$

The reduction from 1.27 μs to 1.14 μs is small because the latency term is now dominant — message size has halved but the floor stays at 1 μs.

*Concepts: [Interconnects and Network Topologies](../concepts/Interconnects_and_Network_Topologies.md), [Domain Decomposition Overheads](../concepts/Domain_Decomposition_Overheads.md)*

### 3(b) Memory Intensive Service — 452 nodes, 12 656 cores, 512 GB / node

Cores per node: $12656 / 452 = 28$.

#### (i) Largest cores available to an OpenMP program (3 marks)

OpenMP is a **shared-memory** programming model; threads share the address space of one process. On a cluster, that process is confined to a single compute node, so OpenMP cannot reach beyond one node's cores.

**Maximum: 28 cores** — the cores in one node.

*Concepts: [OpenMP](../concepts/OpenMP.md), [OpenMP vs. MPI](../comparisons/OpenMP_vs_MPI.md)*

#### (ii) Largest cores available to an MPI program (3 marks)

MPI uses **distributed memory** with explicit message passing, so it scales across nodes via the interconnect.

**Maximum: 12 656 cores** — every core on the system can host an MPI rank.

*Concepts: [Message Passing Interface (MPI)](../concepts/Message_Passing_Interface_MPI.md)*

#### (iii) Largest cores available to a Unified Parallel C program (4 marks)

UPC presents a **single shared, partitioned address space** across both shared-memory (SMP, NUMA) *and* distributed-memory (cluster) systems. The Wikipedia description explicitly says it works on clusters via PGAS, with one thread per processor.

So UPC is not confined to one node; it can use the whole machine like MPI.

**Maximum: 12 656 cores.**

(The key word is "partitioned global address space". Each variable is owned by one processor, but any processor can read/write it; the runtime handles the off-node accesses by communication. There is no shared-memory hardware requirement.)

*Concepts: [Cluster Architecture](../concepts/Cluster_Architecture.md), [HPC Programming Languages](../concepts/HPC_Programming_Languages.md)*

### 3(c) Scaling laws

#### (i) Amdahl's Law (3 marks)

$$S_N = \frac{1}{s + \dfrac{p}{N}}$$

where:
- $S_N$ is the speed-up on $N$ processors,
- $s$ is the *serial* fraction of the program (the part that cannot be parallelised),
- $p$ is the *parallel* fraction, with $s + p = 1$,
- $N$ is the number of processors.

Amdahl's Law describes **strong scaling** — the total problem size is held fixed, and we ask how runtime decreases as more processors are thrown at it.

*Concepts: [Parallel Scaling](../concepts/Parallel_Scaling.md)*

#### (ii) Maximum possible speed-up under Amdahl's Law (2 marks)

Take $N \to \infty$:
$$\lim_{N \to \infty} S_N = \frac{1}{s} = \frac{1}{1 - p}$$

The serial fraction is the absolute ceiling — even with infinite processors, the parallel time can never beat $s \cdot T_0$.

#### (iii) Gustafson's Law (3 marks)

$$S_N = s + p \cdot N$$

where $s$ and $p$ are the serial and parallel fractions of the **parallel runtime** (such that $s + p = 1$), and $N$ is the number of processors.

Gustafson's Law describes **weak scaling** — the problem size *grows* with $N$ so that the work per processor stays constant. Under this regime, speed-up grows linearly with $N$ (no Amdahl ceiling) because we are not trying to shrink a fixed serial fraction.

*Concepts: [Parallel Scaling](../concepts/Parallel_Scaling.md)*

#### (iv) Run time vs N for ideal weak scaling (2 marks)

Run time is **constant** (independent of $N$). The principle of weak scaling is to grow the problem in proportion to the number of processors, so each processor still has the same amount of work — and therefore the same wall-clock time — regardless of how many of them are working in parallel.
