## Question 1 (40 marks)

### 1(a) Frontier (June 2023 #1 Top500)

| Field | Value |
|:---|:---|
| Cores | 8,699,904 |
| Processor | AMD Optimized 3rd Generation EPYC 64C 2 GHz |
| Interconnect | Slingshot-11 |
| $R_{max}$ | 1,194.00 PFlop/s |
| $R_{peak}$ | 1,679.82 PFlop/s |

#### (i) LINPACK efficiency (3 marks)

$$\eta = \frac{R_{max}}{R_{peak}} = \frac{1194.00}{1679.82} = 0.7108$$

**LINPACK efficiency ≈ 71.1%.**

*Concepts: [Performance Metrics and Top500](../concepts/Performance_Metrics_and_Top500.md)*

#### (ii) Theoretical peak per core (5 marks)

$$R_{peak,\,core} = \frac{R_{peak}}{N_{cores}} = \frac{1679.82 \times 10^{15}\ \text{flop/s}}{8\,699\,904}$$

$$= 1.9309 \times 10^{11}\ \text{flop/s} \approx 193\ \text{GFlop/s}$$

Cross-check using the formula $R_{peak,\,core} = R_{clock} \cdot N_{ops/cycle}$:
$$N_{ops/cycle} = \frac{193 \times 10^9}{2 \times 10^9} \approx 96\ \text{ops/cycle}$$

This is consistent with AMD EPYC supporting 256-bit AVX2 with FMA at high issue width: the EPYC `Trento`/`Genoa` family can deliver up to ~16 double-precision FMAs per cycle per core × dual-issue across multiple FPUs — broadly consistent with ~96 ops/cycle when both pipes plus FMA are counted.

(For exam purposes the key derivation is the division: **≈ 193 GFlop/s per core**.)

*Concepts: [Performance Metrics and Top500](../concepts/Performance_Metrics_and_Top500.md), [Floating Point Arithmetic](../concepts/Floating_Point_Arithmetic.md)*

### 1(b) Parallelisability of two loops

#### (i) `a[i] = 0.2 * a[i]` (4 marks)

```c
for (int i=1; i<N; i++){
    a[i] = 0.2 * a[i];
}
```

**Yes, this loop can be safely parallelised.** Each iteration `i` reads only `a[i]` and writes only `a[i]`. There is no dependency between iterations — iteration 5 doesn't need anything iteration 4 produced. Each thread writes a distinct element, so there's no data race.

```c
#pragma omp parallel for default(none) shared(a) firstprivate(N)
for (int i=1; i<N; i++){
    a[i] = 0.2 * a[i];
}
```

*Concepts: [Data Dependencies and Data Races](../concepts/Data_Dependencies.md), [Parallel Loops in OpenMP](../concepts/Parallel_Loops_OpenMP.md)*

#### (ii) `b[i] = 0.25 * (a[i+1] - a[i])` (4 marks)

```c
for (i=0; i<N-1; i++){
    b[i] = 0.25 * (a[i+1] - a[i]);
}
```

**Yes, this loop can be safely parallelised.** Each iteration:
- *Reads* `a[i+1]` and `a[i]` — no writes to `a` occur, so reading from a neighbouring index is safe.
- *Writes* only `b[i]` — distinct from every other iteration.

There is no loop-carried dependency: nothing in iteration `i` reads or writes a value that iteration `i+1` produces.

```c
#pragma omp parallel for default(none) shared(a, b) firstprivate(N)
for (int i=0; i<N-1; i++){
    b[i] = 0.25 * (a[i+1] - a[i]);
}
```

(Note: this would *not* be parallelisable if it were `a[i] = 0.25 * (a[i+1] - a[i])` — that would create an anti-dependency on `a` across iterations.)

*Concepts: [Data Dependencies and Data Races](../concepts/Data_Dependencies.md)*

### 1(c) Top500

#### (i) LINPACK workload type (2 marks)

LINPACK uses a **dense linear-algebra** workload — solving $Ax = b$ via LU factorisation on a dense matrix. It is BLAS Level 3 (matrix-matrix) work with high arithmetic intensity, so it is heavily compute-bound and achieves a high fraction of $R_{peak}$.

*Concepts: [Performance Metrics and Top500](../concepts/Performance_Metrics_and_Top500.md), [BLAS and Dense Matrices](../concepts/BLAS_and_Dense_Matrices.md)*

#### (ii) "All HPC will be commodity clusters" — for OR against (6 marks)

(Same question structure as ECM3446-23May Q1(a)(iii); see that file for the long form. Summary:)

**For:**
- Most of the Top500 below the top tier *is* already x86 + Infiniband or Slingshot — and the trend has been increasing for two decades.
- Off-the-shelf parts give the best price/performance for the typical HPC user.
- Open-source HPC software (Linux, OpenMPI, Slurm) reduces the cost of building custom systems.
- Cloud providers offering HPC services use commodity-like configurations.

**Against:**
- The top of the list is still dominated by MPPs with custom interconnects (Slingshot-11 on Frontier, Tofu-D on Fugaku, IBM custom on the BlueGene line).
- Future exascale systems will need bespoke power, cooling and network design that off-the-shelf assembly cannot deliver.
- Specialised AI/ML training systems (e.g. NVIDIA DGX SuperPOD, Cerebras) are pulling architecture *away* from generic commodity.
- Custom interconnects achieve sub-microsecond latency that commodity Ethernet cannot.

A defensible argument either way scores. Mark scheme typically rewards balanced evidence.

*Concepts: [Cluster Architecture](../concepts/Cluster_Architecture.md), [Interconnects and Network Topologies](../concepts/Interconnects_and_Network_Topologies.md)*

### 1(d) The buggy reduction

```c
int sum = 0;
#pragma omp parallel for default(none) shared(sum)
for (int i=1; i<N+1; i++){
    sum += i;
}
```

#### (i) Why are results incorrect? (6 marks)

`sum` is declared `shared` and is updated by every thread. The statement `sum += i` is **not atomic** — it expands to:
1. Load the current value of `sum` from memory.
2. Add `i` to it.
3. Store the result back to memory.

When multiple threads execute these three steps simultaneously, two threads can both load the same value of `sum`, both add their `i`, and both store back — losing one of the contributions. This is a textbook **data race**.

The error is non-deterministic: the answer changes between runs and is generally less than the true value (123 + 456 + 789 ≠ correct because some additions are dropped). For this loop the correct value is $\sum_{i=1}^{100} i = 5050$, but the parallel run will print a smaller, run-dependent number.

*Concepts: [Data Dependencies and Data Races](../concepts/Data_Dependencies.md), [Variable Scoping in OpenMP](../concepts/Variable_Scoping_OpenMP.md)*

#### (ii) Modify the program for correct results (2 marks)

Replace `shared(sum)` with `reduction(+ : sum)`:

```c
int sum = 0;
#pragma omp parallel for default(none) reduction(+:sum)
for (int i=1; i<N+1; i++){
    sum += i;
}
```

The `reduction` clause gives each thread a private copy of `sum` initialised to the addition identity (0), each thread accumulates into its copy without interfering with the others, and at the end of the parallel region the runtime combines the partials into the final shared `sum` atomically.

*Concepts: [Variable Scoping in OpenMP](../concepts/Variable_Scoping_OpenMP.md)*

### 1(e) Load balancing (8 marks total)

#### (i) Definition of parallel efficiency (2 marks)

$$E_N = \frac{S_N}{N} = \frac{T_0}{N \cdot T_N}$$

Speed-up per processor — perfect linear scaling gives $E_N = 1$.

*Concepts: [Parallel Scaling](../concepts/Parallel_Scaling.md)*

#### (ii) Load balancing (4 marks)

Distribution of work across parallel workers so they all finish at roughly the same time. Poor load balance → fast workers idle at the next barrier waiting for the slow ones (starvation), reducing efficiency. The slowest worker dictates total runtime, so unbalanced workloads waste hardware and money.

*Concepts: [Load Balancing and Scheduling](../concepts/Load_Balancing_and_Scheduling.md)*

#### (iii) Improving load balancing in OpenMP loops (2 marks)

Use the `schedule` clause:
- `schedule(dynamic, chunk)` — threads claim chunks at runtime, balancing irregular work.
- `schedule(guided, chunk)` — exponentially-shrinking chunk sizes; lower overhead than dynamic.

The default `schedule(static)` gives equal-size chunks at compile time and balances poorly when iterations have non-uniform cost.

*Concepts: [Load Balancing and Scheduling](../concepts/Load_Balancing_and_Scheduling.md)*

## Question 2 (30 marks)

### 2(a) Memory Intensive Service (Durham) — 452 nodes, 12 656 cores

Cores per node: $12656 / 452 = 28$.

#### (i) OpenMP largest core count (2 marks)

OpenMP is shared-memory; one OpenMP program is one process whose threads share an address space, so it cannot reach across nodes. **Maximum: 28 cores** (one node).

#### (ii) MPI largest core count (2 marks)

MPI uses distributed memory and explicit messaging across nodes. **Maximum: 12 656 cores** (every core on the system).

#### (iii) UPC largest core count (2 marks)

The Wikipedia description states UPC supports both shared and distributed-memory machines via PGAS, with one thread per processor. UPC therefore scales across the cluster like MPI. **Maximum: 12 656 cores.**

*Concepts: [OpenMP](../concepts/OpenMP.md), [Message Passing Interface (MPI)](../concepts/Message_Passing_Interface_MPI.md), [HPC Programming Languages](../concepts/HPC_Programming_Languages.md)*

### 2(b) Strong scaling, target speed-up 100

(Same calculations as ECM3446-23May Q3(a) — included here in full.)

#### (i) Smallest p that allows S = 100 (3 marks)

$$\frac{1}{1-p} \ge 100 \implies p \ge 0.99$$

#### (ii) N for p = 0.999 to give S = 100 (3 marks)

$$100 = \frac{1}{0.001 + 0.999/N} \implies \frac{0.999}{N} = 0.009 \implies N = 111$$

#### (iii) Smallest N with overhead V/T₀ = 0.005 (4 marks)

$$100 = \frac{1}{0.001 + 0.999/N + 0.005} \implies \frac{0.999}{N} = 0.004 \implies N = 249.75$$

Round up: **N = 250 processors.**

### 2(c) Halo exchange (8192 × 8192 domain)

#### (i) Values per halo, 8 × 8 decomposition (2 marks)

Sub-domain side: $8192 / 8 = 1024$. Halo length = 1024.

#### (ii) Bytes per halo (2 marks)

$1024 \times 8 = 8192$ bytes.

#### (iii) Transmission time, L = 1 µs, B = 12.5 GB/s (3 marks)

$$t = 1.0 \times 10^{-6} + \frac{8192}{12.5 \times 10^9} = 1.0 \times 10^{-6} + 6.554 \times 10^{-7}$$
$$\boxed{t \approx 1.66 \times 10^{-6}\ \text{s} = 1.66\ \mu\text{s}}$$

*Concepts: [Interconnects and Network Topologies](../concepts/Interconnects_and_Network_Topologies.md)*

#### (iv) Transmission time, 16 × 16 decomposition (2 marks)

Sub-domain $512 \times 512$, halo length 512, bytes $512 \times 8 = 4096$.

$$t = 1.0 \times 10^{-6} + \frac{4096}{12.5 \times 10^9} = 1.0 \times 10^{-6} + 3.277 \times 10^{-7}$$
$$\boxed{t \approx 1.33 \times 10^{-6}\ \text{s} = 1.33\ \mu\text{s}}$$

#### (v) Factor by which the comm/comp ratio increased (3 marks)

- **Communication time** changed by factor $\frac{1.33}{1.66} = 0.799$ (got slightly faster).
- **Computation per sub-domain** scales as the area: $1024^2$ in 8×8 vs $512^2$ in 16×16, ratio $\left(\frac{512}{1024}\right)^2 = \frac{1}{4}$ — computation got 4× faster.

The ratio is comm/comp:
$$\frac{(\text{comm/comp})_{16}}{(\text{comm/comp})_{8}} = \frac{t_{comm,16} / t_{comp,16}}{t_{comm,8} / t_{comp,8}} = \frac{1.33 / 1.66}{1/4} = 0.799 \times 4 \approx 3.20$$

**The comm/comp ratio increased by a factor of ≈ 3.2.**

*Concepts: [Domain Decomposition Overheads](../concepts/Domain_Decomposition_Overheads.md)*

#### (vi) Why comm/comp doesn't scale as halo/sub-domain ratio (2 marks)

If comm time were purely bandwidth-limited, communication time would scale linearly with halo size (which scales as $1/N$, where $N$ is the side length). It would then track the surface-to-volume ratio of the sub-domain. But communication time also includes a **fixed latency $L$** that is independent of message size: $t = L + M/B$.

For small halo messages the latency term dominates. As the sub-domain shrinks, the message-size term shrinks proportionally but $L$ stays the same — so communication time stops shrinking, while computation continues to drop with $N^2$. The actual comm/comp ratio therefore grows *faster* than the simple $\propto 1/N$ surface-to-volume prediction.

*Concepts: [Interconnects and Network Topologies](../concepts/Interconnects_and_Network_Topologies.md), [Domain Decomposition Overheads](../concepts/Domain_Decomposition_Overheads.md)*

## Question 3 (30 marks)

### 3(a) Compute node selection

(Same compute-node comparison as ECM3446-23May Q3(b).)

- **Type 1:** 2 × 64 cores × 2.2 GHz × 4 ops/cycle
- **Type 2:** 2 × 28 cores × 2.8 GHz × 16 ops/cycle

#### (i) Higher peak performance? (3 marks)

$$R_{peak,\,1} = 2 \times 64 \times 2.2 \times 4 = 1126.4\ \text{GFlop/s}$$
$$R_{peak,\,2} = 2 \times 28 \times 2.8 \times 16 = 2508.8\ \text{GFlop/s}$$

**Type 2 is higher.**

#### (ii) Application doesn't use vectors / FMA — does that change the choice? (3 marks)

**Yes — switch to Type 1.** Type 2's lead comes entirely from its 16 ops/cycle figure (wide AVX with FMA). If the application can only issue scalar instructions, the effective ops/cycle drops toward ~1 for both nodes, and the comparison reduces to (cores × clock):

- Type 1 effective: $2 \times 64 \times 2.2 \approx 282$ GHz·cores
- Type 2 effective: $2 \times 28 \times 2.8 \approx 157$ GHz·cores

Type 1 wins by core count.

#### (iii) Sparse matrix workload — what extra info do you need? (2 marks)

**Memory bandwidth.** Sparse operations have low arithmetic intensity (≈ $O(1)$, with indirect addressing through `col_ind`/`row_ptr`), so they are memory-bound. The peak FLOP rate is irrelevant; performance is set by how fast the node can stream data from DRAM.

*Concepts: [Sparse Matrices and CSR](../concepts/Sparse_Matrices_and_CSR.md), [Arithmetic Intensity and Roofline Model](../concepts/Arithmetic_Intensity_and_Roofline_Model.md)*

### 3(c) Diffusion equation

Time step constraint: $\Delta t = \dfrac{1}{4} \dfrac{(\Delta x)^2}{K}$.

#### (i) 1-D N grid points vs 2-D N × N grid points (2 marks)

Grid points scale by $N$ (1-D) vs $N^2$ (2-D). With identical spatial resolution and same number of time steps, the work per step is proportional to grid points, so:

**The 2-D calculation costs $N$ times more than the 1-D calculation.**

(Each time step touches $N^2$ points instead of $N$.)

*Concepts: [Diffusion Equation](../concepts/Diffusion_Equation.md), [Finite Difference Method](../concepts/Finite_Difference_Method.md)*

#### (ii) Cost of 2-D calc when Δx is reduced by 10× (5 marks)

Three things change when $\Delta x \to \Delta x / 10$ at fixed end time and domain size:

1. **Grid points per dimension** scale as $1/\Delta x$, so they grow by ×10. In 2-D the total grid points grow by $10 \times 10 = 100$.
2. **Time step** scales as $(\Delta x)^2$, so $\Delta t$ shrinks by $10^2 = 100$.
3. **Number of time steps** to reach the same end time grows by ×100.

Total cost is `points × time steps`:
$$\text{factor} = 100 \times 100 = 10\,000$$

**Cost grows by a factor of 10 000.** (A 10× refinement is enormously expensive in 2-D diffusion.)

*Concepts: [Diffusion Equation](../concepts/Diffusion_Equation.md), [Numerical Stability and CFL](../concepts/Numerical_Stability_and_CFL.md)*

#### (iii) Comparison with CFL on the advection equation (2 marks)

The CFL condition for advection: $\Delta t \le C \frac{\Delta x}{v_{max}}$ — linear in $\Delta x$.

The diffusion stability condition: $\Delta t \le \frac{1}{4} \frac{(\Delta x)^2}{K}$ — *quadratic* in $\Delta x$.

When $\Delta x$ shrinks (finer grid), the advection time step shrinks linearly while the diffusion time step shrinks quadratically. **The diffusion solver is far more expensive at fine resolutions** — refining the grid by 10× costs 10× more for advection but 100× more for diffusion (per time step), because diffusion needs 10× as many time steps in addition to its 10× extra spatial work.

*Concepts: [Numerical Stability and CFL](../concepts/Numerical_Stability_and_CFL.md), [Advection Equation](../concepts/Advection_Equation.md), [Diffusion Equation](../concepts/Diffusion_Equation.md)*

#### (iv) "Unconditionally stable" implicit time stepping (3 marks)

**Unconditionally stable** means errors do not grow without bound for *any* choice of $\Delta t$, regardless of $\Delta x$ — the explicit stability constraint is removed entirely. Implicit methods (like backward Euler or Crank-Nicolson) achieve this by solving a linear system at each step that couples all grid points together.

But $\Delta t$ still cannot be arbitrarily large because:
- **Truncation (accuracy) error grows with $\Delta t$.** Stability is a separate question from accuracy: a stable but inaccurate scheme can still produce nonsense if the time step is chosen too coarse to resolve the physical dynamics.
- The discrete solution can drift far from the true solution, miss transients, and over-smooth features.
- Practical $\Delta t$ is bounded by acceptable error, not by stability.

So implicit methods buy *stability* freedom, but accuracy still requires reasonable $\Delta t$.

*Concepts: [Numerical Stability and CFL](../concepts/Numerical_Stability_and_CFL.md), [Diffusion Equation](../concepts/Diffusion_Equation.md), [Finite Difference Method](../concepts/Finite_Difference_Method.md)*
