### 1(b) For each of the following loops (written in the C programming language) say whether it can be parallelised by adding OpenMP directives and still give the correct result, and explain why. You may assume that the loop gives correct results without OpenMP directives.

#### (i) *(4 marks)*

```c
for (int i=1; i<N; i++){
    a[i]=0.2*a[i];
}
```

**Answer:**

**Yes, this loop can be safely parallelised.** Each iteration `i` reads only `a[i]` and writes only `a[i]`. There is no dependency between iterations — iteration 5 doesn't need anything iteration 4 produced. Each thread writes a distinct element, so there's no data race.

```c
#pragma omp parallel for default(none) shared(a) firstprivate(N)
for (int i=1; i<N; i++){
    a[i] = 0.2 * a[i];
}
```

*Concepts: [Data Dependencies and Data Races](../concepts/Data_Dependencies.md), [Parallel Loops in OpenMP](../concepts/Parallel_Loops_OpenMP.md)*

---

#### (ii) *(4 marks)*

```c
for (i=0; i<N-1; i++){
    b[i]=0.25*(a[i+1]-a[i]);
}
```

**Answer:**

**Yes, this loop can be safely parallelised.** Each iteration:
- *Reads* `a[i+1]` and `a[i]` — no writes to `a` occur, so reading from a neighbouring index is safe.
- *Writes* only `b[i]` — distinct from every other iteration.

There is no loop-carried dependency: nothing in iteration `i` reads or writes a value that iteration `i+1` produces.

```c
#pragma omp parallel for default(none) shared(a, b) firstprivate(N)
for (int i=0; i<N-1; i++){
    b[i] = 0.25 * (a[i+1] - a[i]);
}
```

*(Note: this would NOT be parallelisable if it were `a[i] = 0.25*(a[i+1]-a[i])` — that creates an anti-dependency on `a` across iterations.)*

*Concepts: [Data Dependencies and Data Races](../concepts/Data_Dependencies.md)*

### 1(c) The Top500 list ranks the fastest supercomputers in the world according to the performance on the Linpack benchmark.

#### (i) What type of workload does the LINPACK benchmark use? *(2 marks)*

**Answer:**

LINPACK uses a **dense linear-algebra** workload — solving $Ax = b$ via LU factorisation on a dense matrix. It is BLAS Level 3 (matrix-matrix) work with high arithmetic intensity, so it is heavily compute-bound and achieves a high fraction of $R_{peak}$.

*Concepts: [Performance Metrics and Top500](../concepts/Performance_Metrics_and_Top500.md), [BLAS and Dense Matrices](../concepts/BLAS_and_Dense_Matrices.md)*

**Answer:**

**For:**
- Most of the Top500 below the top tier *is* already x86 + Infiniband or Slingshot — and the trend has been increasing for two decades.
- Off-the-shelf parts give the best price/performance for the typical HPC user.

### 1(d) The following C program produces incorrect results when OpenMP is used:

```c
#include <stdio.h>

const int N=100;

int main(){
    int sum=0;
#pragma omp parallel for default(none) shared(sum)
    for (int i=1; i<N+1; i++){
        sum+=i;
    }
    printf("Sum= %d\n", sum);
    return 0;
}
```

#### (i) Why are the results incorrect when OpenMP is used? You may assume that the results are correct when OpenMP is not used. *(6 marks)*

**Answer:**

`sum` is declared `shared` and is updated by every thread. The statement `sum += i` is **not atomic** — it expands to:
1. Load the current value of `sum` from memory.
2. Add `i` to it.
3. Store the result back to memory.

When multiple threads execute these three steps simultaneously, two threads can both load the same value of `sum`, both add their `i`, and both store back — losing one of the contributions. This is a textbook **data race**.

The error is non-deterministic: the answer changes between runs and is generally less than the true value. The correct value is $\sum_{i=1}^{100} i = 5050$, but the parallel run will print a smaller, run-dependent number.

*Concepts: [Data Dependencies and Data Races](../concepts/Data_Dependencies.md), [Variable Scoping in OpenMP](../concepts/Variable_Scoping_OpenMP.md)*

---

#### (ii) Modify the program so that it produces correct results. *(2 marks)*

**Answer:**

Replace `shared(sum)` with `reduction(+ : sum)`:

```c
int sum = 0;
#pragma omp parallel for default(none) reduction(+:sum)
for (int i=1; i<N+1; i++){
    sum += i;
}
```

The `reduction` clause gives each thread a private copy of `sum` initialised to 0, each thread accumulates into its copy without interference, and at the end the runtime combines the partials atomically.

*Concepts: [Variable Scoping in OpenMP](../concepts/Variable_Scoping_OpenMP.md)*

### 1(e) Effective load balancing can significantly improve the performance of a parallel program.

#### (i) Define what is meant by parallel efficiency? *(2 marks)*

**Answer:**

$$E_N = \frac{S_N}{N} = \frac{T_0}{N \cdot T_N}$$

Speed-up per processor — perfect linear scaling gives $E_N = 1$.

*Concepts: [Parallel Scaling](../concepts/Parallel_Scaling.md)*

---

#### (ii) Explain what is meant by "load balancing". *(4 marks)*

**Answer:**

Distribution of work across parallel workers so they all finish at roughly the same time. Poor load balance → fast workers idle at the next barrier waiting for the slow ones (starvation), reducing efficiency. The slowest worker dictates total runtime, so unbalanced workloads waste hardware and money.

*Concepts: [Load Balancing and Scheduling](../concepts/Load_Balancing_and_Scheduling.md)*

---

#### (iii) How can load balancing be improved when parallelising a loop with OpenMP? *(2 marks)*

**Answer:**

Use the `schedule` clause:
- `schedule(dynamic, chunk)` — threads claim chunks at runtime, balancing irregular work.
- `schedule(guided, chunk)` — exponentially-shrinking chunk sizes; lower overhead than dynamic.

The default `schedule(static)` gives equal-size chunks at compile time and balances poorly when iterations have non-uniform cost.

*Concepts: [Load Balancing and Scheduling](../concepts/Load_Balancing_and_Scheduling.md)*

### 2(a) The "Memory Intensive Service" is an HPC facility hosted by the University of Durham at the Institute for Computational Cosmology (ICC). The system comprises 452 compute nodes, each with 512 GB of RAM, and offering a total of 12 656 cores. All the compute nodes have the same number of processor cores.

Based on this information what is the largest number of processor cores available to a Unified Parallel C program running on this system and why? *(2 marks)*

### 2(b) You have been given the task of parallelising a program and need to achieve a strong scaling speed-up of at least 100.

#### (i) What is the smallest parallel fraction which permits a speed-up of 100? *(3 marks)*

**Answer:**

By Amdahl's Law, maximum speed-up = $\frac{1}{1-p}$. We need:
$$\frac{1}{1-p} \ge 100 \implies p \ge 0.99$$

**Smallest parallel fraction: p = 0.99.**

---

#### (ii) You parallelise the program and achieve a parallel fraction of p = 0.999. How many processors are required to obtain a speed up of 100 assuming idealised strong scaling? *(3 marks)*

**Answer:**

$$100 = \frac{1}{0.001 + 0.999/N} \implies \frac{0.999}{N} = 0.009 \implies N = 111$$

**N = 111 processors.**

---

#### (iii) If the total parallel overhead is equivalent to 0.5% of the serial run time, what is the smallest number of processors required to achieve a speed up of at least 100? *(4 marks)*

**Answer:**

$$100 = \frac{1}{0.001 + 0.999/N + 0.005} \implies \frac{0.999}{N} = 0.004 \implies N = 249.75$$

Round up: **N = 250 processors.**

*Concepts: [Parallel Scaling](../concepts/Parallel_Scaling.md)*

### 2(c) Consider a two-dimensional PDE solver (same stencil as 2(a)) on an 8192 × 8192 domain with halos stored on each sub-domain.

#### (i) The calculation is domain decomposed in an 8 × 8 configuration such that there are 64 square sub-domains of equal size. How many values need to be exchanged when one halo is sent to one neighbouring process? *(2 marks)*

**Answer:**

Sub-domain side: $8192 / 8 = 1024$. **Halo length = 1024 values.**

---

#### (ii) How many bytes need to be sent if the values are represented using IEEE754 double precision? *(2 marks)*

**Answer:**

$1024 \times 8 = \mathbf{8192}$ **bytes.**

---

#### (iii) What is the transmission time for sending a single halo over a network with a latency of 1µs (10⁻⁶s) and a bandwidth of 12.5 GB/s (12.5 gigabytes per second)? Show your working. *(3 marks)*

**Answer:**

$$t = 1.0 \times 10^{-6} + \frac{8192}{12.5 \times 10^9} = 1.0 \times 10^{-6} + 6.554 \times 10^{-7}$$
$$\boxed{t \approx 1.655 \times 10^{-6}\ \text{s} = 1.655\ \mu\text{s}}$$

*Concepts: [Interconnects and Network Topologies](../concepts/Interconnects_and_Network_Topologies.md)*

---

#### (iv) The domain decomposition is modified so that the domain is decomposed in a 16 × 16 configuration such that there are 256 square sub-domains of equal size. What is the transmission time for one halo with the modified domain decomposition? *(2 marks)*

**Answer:**

Sub-domain $512 \times 512$, halo length 512, bytes $512 \times 8 = 4096$.

$$t = 1.0 \times 10^{-6} + \frac{4096}{12.5 \times 10^9} = 1.0 \times 10^{-6} + 3.277 \times 10^{-7}$$
$$\boxed{t \approx 1.328 \times 10^{-6}\ \text{s} = 1.328\ \mu\text{s}}$$

---

#### (v) Modifying the domain decomposition has changed the ratio of communication time to computation time. By what factor has the ratio of communication time to computation time increased? *(3 marks)*

**Answer:**

- **Communication time ratio:** $t_{16} / t_{8} = 1.328 / 1.655 \approx 0.802$ (comm got slightly *faster*)
- **Computation per sub-domain** scales as area: $1024^2$ (8×8) vs $512^2$ (16×16), so computation fell by factor $\left(\frac{512}{1024}\right)^2 = \frac{1}{4}$

The comm/comp ratio:
$$\frac{(\text{comm/comp})_{16}}{(\text{comm/comp})_{8}} = \frac{t_{comm,16} / t_{comp,16}}{t_{comm,8} / t_{comp,8}} = 0.802 \times 4 \approx 3.2$$

**The comm/comp ratio increased by a factor of ≈ 3.2.**

*Concepts: [Domain Decomposition Overheads](../concepts/Domain_Decomposition_Overheads.md)*

---

#### (vi) Why is the ratio of communication time to computation time not scaling as the number of grid points in the halo divided by the number of grid points in the sub-domain? *(2 marks)*

**Answer:**

If comm time were purely bandwidth-limited, communication time would scale linearly with halo size (proportional to $1/N$ where $N$ is the decomposition side). But communication time also includes a **fixed latency $L$** independent of message size: $t = L + M/B$.

For small halo messages the latency term dominates. As the sub-domain shrinks, the message-size term shrinks proportionally but $L$ stays the same — so communication time stops shrinking as fast as halo size, while computation continues to drop with $N^2$. The actual comm/comp ratio therefore grows *faster* than the simple surface-to-volume prediction.

*Concepts: [Interconnects and Network Topologies](../concepts/Interconnects_and_Network_Topologies.md), [Domain Decomposition Overheads](../concepts/Domain_Decomposition_Overheads.md)*

## Question 3 (30 marks)

### 3(a) You are given the task of planning an upgrade to an HPC cluster which involves replacing all the compute nodes. The total number of compute nodes will not change and there will be no other changes to the hardware. You need to choose between two compute node specifications:

- **Compute node type 1:** number of sockets=2, number of cores per socket=64, processor clock frequency=2.2 GHz, floating point operations per cycle=4
- **Compute node type 2:** number of sockets=2, number of cores per socket=28, processor clock frequency=2.8 GHz, floating point operations per cycle=16

The cost of the compute nodes is the same so you need to make a choice based on performance.

#### (i) Which type of compute node has the higher theoretical peak performance? Show your working. *(3 marks)*

**Answer:**

$$R_{peak,1} = 2 \times 64 \times 2.2 \times 4 = 1126.4\ \text{GFlop/s}$$
$$R_{peak,2} = 2 \times 28 \times 2.8 \times 16 = 2508.8\ \text{GFlop/s}$$

**Type 2 has the higher theoretical peak performance.**

*Concepts: [Performance Metrics and Top500](../concepts/Performance_Metrics_and_Top500.md)*

---

#### (ii) You know that the application which will run on the system has not been optimised to make effective use of the instruction set in modern processors. This means that the application is not able to use vector instructions and fused-multiply-add instructions. Explain whether this would influence your choice of compute node and why. *(3 marks)*

**Answer:**

**Yes — switch to Type 1.** Type 2's lead comes entirely from its 16 ops/cycle figure (wide AVX with FMA). If the application can only issue scalar instructions, effective ops/cycle drops toward ~1 for both nodes, and the comparison reduces to (cores × clock):

- Type 1 effective: $2 \times 64 \times 2.2 \approx 282\ \text{GHz·cores}$
- Type 2 effective: $2 \times 28 \times 2.8 \approx 157\ \text{GHz·cores}$

Type 1 wins by core count for an unoptimised scalar code.

*Concepts: [Floating Point Arithmetic](../concepts/Floating_Point_Arithmetic.md)*

#### (iii) You know that the application which will run on the system makes extensive use of sparse matrix operations. What extra piece of information would you need to know in order to assess the suitability of the compute nodes for running this application and why is this information important? *(2 marks)*

**Answer:**

**Memory bandwidth.** Sparse operations have low arithmetic intensity ($O(1)$, with indirect addressing), so they are memory-bound. Performance is set by how fast the node can stream data from DRAM, not by FLOP rate.

### 3(c) A finite difference method is used to solve the diffusion equation. The time step used to compute the solution is calculated according to the constraint:

$$\Delta t = \frac{1}{4} \frac{(\Delta x)^2}{K}$$

where $\Delta t$ is the time step, $\Delta x$ is the spacing between grid points, and $K$ is a constant.

#### (i) How does the computational cost of calculating a one-dimensional solution with N grid points compare with the computational cost of calculating a two-dimensional solution with N grid points in each dimension? You should assume that spatial resolution and number of time steps calculated are the same in both cases. Explain your reasoning. *(2 marks)*

**Answer:**

Grid points scale by $N$ (1-D) vs $N^2$ (2-D). With identical spatial resolution and same number of time steps, the work per step is proportional to grid points, so:

**The 2-D calculation costs $N$ times more than the 1-D calculation.**

*Concepts: [Diffusion Equation](../concepts/Diffusion_Equation.md), [Finite Difference Method](../concepts/Finite_Difference_Method.md)*

---

#### (ii) How does the computational cost of the two-dimensional calculation increase if Δx is reduced by a factor of 10? You should assume that end time of the calculation and the extent of the computational domain remain constant. Explain your reasoning. *(5 marks)*

**Answer:**

Three things change when $\Delta x \to \Delta x / 10$ at fixed end time and domain size:

1. **Grid points per dimension** scale as $1/\Delta x$, so they grow by ×10. In 2-D the total grid points grow by $10 \times 10 = 100$.
2. **Time step** scales as $(\Delta x)^2$, so $\Delta t$ shrinks by $10^2 = 100$.
3. **Number of time steps** to reach the same end time grows by ×100.

Total cost is `points × time steps`:
$$\text{factor} = 100 \times 100 = \mathbf{10\,000}$$

**Cost grows by a factor of 10 000.** A 10× spatial refinement is enormously expensive in 2-D diffusion.

*Concepts: [Diffusion Equation](../concepts/Diffusion_Equation.md), [Numerical Stability and CFL](../concepts/Numerical_Stability_and_CFL.md)*

---

#### (iii) How does the time step constraint for the diffusion equation compare to the CFL condition for the advection equation and what effect does this have on the computational cost? *(2 marks)*

**Answer:**

- **CFL (advection):** $\Delta t \le C \frac{\Delta x}{v_{max}}$ — **linear** in $\Delta x$.
- **Diffusion stability:** $\Delta t \le \frac{1}{4} \frac{(\Delta x)^2}{K}$ — **quadratic** in $\Delta x$.

When $\Delta x$ shrinks, the diffusion time step shrinks quadratically while the advection time step shrinks linearly. **The diffusion solver is far more expensive at fine resolutions** — refining the grid by 10× costs 10× more for advection (per unit time) but 100× more for diffusion, because diffusion requires 10× as many time steps *in addition* to its 10× extra spatial work.

*Concepts: [Numerical Stability and CFL](../concepts/Numerical_Stability_and_CFL.md), [Advection Equation](../concepts/Advection_Equation.md), [Diffusion Equation](../concepts/Diffusion_Equation.md)*

---

#### (iv) The diffusion equation can be solved using implicit time stepping which is unconditionally stable. What is meant by "unconditionally stable" and why can the time step not be arbitrarily large? *(3 marks)*

**Answer:**

**Unconditionally stable** means errors do not grow without bound for *any* choice of $\Delta t$, regardless of $\Delta x$ — the explicit stability constraint is removed entirely. Implicit methods (like backward Euler or Crank-Nicolson) achieve this by solving a linear system at each step that couples all grid points together.

But $\Delta t$ still cannot be arbitrarily large because:
- **Truncation (accuracy) error grows with $\Delta t$.** Stability is a separate question from accuracy: a stable but inaccurate scheme can still produce nonsense if the time step is too coarse to resolve the physical dynamics.
- The discrete solution can drift far from the true solution, miss transients, and over-smooth features.
- Practical $\Delta t$ is bounded by acceptable error, not by stability.

So implicit methods buy *stability* freedom, but accuracy still requires reasonable $\Delta t$.

*Concepts: [Numerical Stability and CFL](../concepts/Numerical_Stability_and_CFL.md), [Diffusion Equation](../concepts/Diffusion_Equation.md), [Finite Difference Method](../concepts/Finite_Difference_Method.md)*
