## Question 1 (40 marks)

### 1(a) The Top500 list ranks the fastest supercomputers in the world and has been running since 1993.

#### (i) What is the performance benchmark used for the Top500 list? *(2 marks)*

**Answer:**

The **High-Performance Linpack (HPL)** benchmark — also written as LINPACK. The Top500 list ranks systems by their measured maximum HPL performance ($R_{max}$, in flop/s).

*Concepts: [Performance Metrics and Top500](../concepts/Performance_Metrics_and_Top500.md)*

---

#### (ii) What type of workload does the performance benchmark for Top500 list use? *(2 marks)*

**Answer:**

HPL solves a **dense system of linear equations** $Ax = b$ using LU factorisation. Because it is a dense matrix operation (BLAS Level 3, like DGEMM), it has high arithmetic intensity ($\sim O(N)$) and is heavily **compute-bound** — most systems achieve a high fraction of $R_{peak}$ on it.

*Concepts: [BLAS and Dense Matrices](../concepts/BLAS_and_Dense_Matrices.md), [Arithmetic Intensity and Roofline Model](../concepts/Arithmetic_Intensity_and_Roofline_Model.md)*

---

#### (iii) "Given current trends soon all High Performance Computing systems are going to be commodity clusters." Argue for OR against this position. *(4 marks)*

**Answer:**

**For (the case that the trend is real):**
- Commodity clusters use off-the-shelf processors, motherboards and Ethernet/Infiniband interconnects → cost-effective and procurable independently.
- A growing share of the Top500 list is dominated by commodity-style x86 + Infiniband systems.
- Open-source software stacks (Linux, OpenMPI, Slurm) lower the barrier to building large clusters.
- Commercial cloud providers offer HPC instances built on commodity parts.

**Against (the case the trend is overstated):**
- The very top of the Top500 list is still dominated by **Massively Parallel Processors (MPPs)** with custom interconnects (e.g. Cray Slingshot, Fujitsu Tofu, IBM BlueGene). Custom interconnects deliver lower latency and higher bisection bandwidth than commodity Ethernet.
- GPU-accelerated and ARM-based machines are pushing the architectural envelope away from pure x86 commodity.
- Power and cooling at the exascale require bespoke engineering that no off-the-shelf assembly delivers.

A well-supported answer either way — backed by the trade-off between cost-effectiveness and peak performance/latency — earns full marks.

*Concepts: [Cluster Architecture](../concepts/Cluster_Architecture.md), [Interconnects and Network Topologies](../concepts/Interconnects_and_Network_Topologies.md)*

### 1(b) Effective load balancing can significantly improve the performance of a parallel program.

#### (i) Define what is meant by parallel efficiency? *(2 marks)*

**Answer:**

Parallel efficiency $E_N$ is the speed-up divided by the number of processors:
$$E_N = \frac{S_N}{N} = \frac{T_0}{N \cdot T_N}$$
where $T_0$ is the serial run time, $T_N$ is the run time on $N$ processors, and $S_N = T_0 / T_N$. An efficiency of 1.0 (100%) is ideal/linear speed-up.

*Concepts: [Parallel Scaling](../concepts/Parallel_Scaling.md)*

---

#### (ii) Explain what is meant by "load balancing". *(4 marks)*

**Answer:**

Load balancing is the distribution of work across parallel workers (threads or MPI processes) such that **each worker takes approximately the same time** to complete its share. If the workload is poorly balanced, fast workers finish their work and sit idle at the next synchronisation point (an implicit barrier in OpenMP, or a collective in MPI), while the slowest worker dictates the overall runtime — this idle time is **starvation** and reduces parallel efficiency. Effective load balancing minimises starvation and maximises useful CPU utilisation.

*Concepts: [Load Balancing and Scheduling](../concepts/Load_Balancing_and_Scheduling.md)*

---

#### (iii) How can load balancing be improved when parallelising a loop with OpenMP? *(2 marks)*

**Answer:**

Use the `schedule` clause on `#pragma omp for`:
- `schedule(dynamic, chunk)` — threads grab a new chunk as soon as they finish their current one, balancing uneven iteration costs at runtime.
- `schedule(guided, chunk)` — chunks start large and shrink as work runs out, giving good balance with lower overhead than `dynamic`.

Default `static` scheduling is cheap but performs poorly when iterations have non-uniform cost.

*Concepts: [Load Balancing and Scheduling](../concepts/Load_Balancing_and_Scheduling.md)*

### 1(c) A C program, which is parallelised using MPI, contains the following code fragment:

```c
if (myRank == 0)
    printf("The result is %d\n", globalSum);
else
    /* Wait for the result to be printed out */
    MPI_Barrier(MPI_COMM_WORLD);
}
```

You know that the variable `myRank` is an integer variable which holds the process's rank within the `MPI_COMM_WORLD` communicator.

#### (i) What happens when the program reaches this section of code and why? *(6 marks)*

**Answer:**

The program **deadlocks (hangs)**. `MPI_Barrier` is a **collective operation**: every process in the communicator (here `MPI_COMM_WORLD`) must call it for any of them to proceed. In this code:
- Rank 0 takes the `if` branch, prints, and **never calls `MPI_Barrier`**.
- All other ranks take the `else` branch and **enter `MPI_Barrier`**, where they wait forever for rank 0 to also enter.

The barrier never completes. The non-zero ranks are blocked indefinitely; the program never reaches `MPI_Finalize` and never terminates cleanly.

*Concepts: [MPI Collective Communication](../concepts/MPI_Collective_Communication.md)*

---

#### (ii) What type of bug is this? *(2 marks)*

**Answer:**

A **deadlock** — specifically, an incorrect use of a collective operation in which not all members of the communicator participate.

### 1(d) Matrix operations are often used in high performance computing to solve systems of linear equations.

#### (i) What is the difference between a sparse matrix and a dense matrix? *(2 marks)*

**Answer:**

- **Dense matrix:** most or all elements are non-zero. Stored as a contiguous 2-D array; access is direct and cache-friendly.
- **Sparse matrix:** the vast majority of elements are zero. Storing every element wastes memory and forces useless multiply-by-zero operations, so specialised formats store only the non-zero entries.

*Concepts: [BLAS and Dense Matrices](../concepts/BLAS_and_Dense_Matrices.md), [Sparse Matrices and CSR](../concepts/Sparse_Matrices_and_CSR.md)*

---

#### (ii) Describe the compressed sparse row format for storing sparse matrices. *(4 marks)*

**Answer:**

CSR uses three 1-D arrays for a matrix with `nrow` rows and `nnz` non-zero entries:

1. **`vals[nnz]`** — the non-zero values, stored row-by-row in row-major order.
2. **`col_ind[nnz]`** — the column index of each value in `vals`.
3. **`row_ptr[nrow + 1]`** — for each row `r`, `row_ptr[r]` gives the index in `vals`/`col_ind` where row `r` starts. The final entry equals `nnz + 1` (1-indexed) or `nnz` (0-indexed) and acts as a sentinel.

This stores $2 \cdot nnz + nrow + 1$ values instead of $nrow \cdot ncol$, saving substantial memory when $nnz \ll nrow \cdot ncol$. The trade-off is **indirect addressing** (looking up `col_ind` to find the column of each value), which is slower than a dense access pattern.

*Concepts: [Sparse Matrices and CSR](../concepts/Sparse_Matrices_and_CSR.md)*

---

#### (iii) How does the number of bytes of data transferred from memory scale with N for dense matrix multiplication and why? *(2 marks)*

**Answer:**

$O(N^2)$. For $C = A \times B$ with $N \times N$ matrices, each matrix has $N^2$ entries. Reading $A$, reading $B$, and writing $C$ all scale as $N^2$, so total data movement is $\sim O(N^2)$ bytes. The number of FLOPs scales as $O(N^3)$, so arithmetic intensity grows as $O(N)$ — DGEMM becomes increasingly compute-bound as $N$ grows.

*Concepts: [BLAS and Dense Matrices](../concepts/BLAS_and_Dense_Matrices.md), [Arithmetic Intensity and Roofline Model](../concepts/Arithmetic_Intensity_and_Roofline_Model.md)*

### 1(e) You are given the task of helping a researcher speed up their program and you suggest parallelising the program. The researcher says: "Parallel programming is too difficult. I've heard that Moores Law is still going. Why can't I get a much faster processor to make my serial program run quicker?" How would you explain to them that they have mis-understood Moores Law and why they can't get a faster single processor core? *(8 marks)*

**Answer:**

The researcher's mistake is conflating **Moore's Law** (a statement about transistor density) with **clock-frequency scaling** (a separate phenomenon that has stopped).

- **Moore's Law:** the number of transistors per chip doubles roughly every two years. This *is* still progressing, though more slowly. It buys us *more transistors* — used to add more cores per chip, not to make a single core faster.
- **Dennard scaling:** the rule that as transistors got smaller, they ran faster and used less power per unit area. Dennard scaling **broke down in the mid-2000s** because of leakage currents, finite gate-oxide thickness, and the resulting power-density wall. Pushing clock frequencies higher pushes power and heat past what air-cooling can dissipate.
- **Consequence:** clock frequencies have been stuck around 2–4 GHz since ~2005. Single-core performance gains now come only from architectural tweaks (wider vector units, better branch prediction, larger caches), not from raw frequency.
- **Implication for HPC:** the only way to get more performance from modern processors is to use the *additional cores* that Moore's Law continues to deliver. That means parallel programming. A serial program leaves all but one core idle and cannot scale with hardware.

In short: Moore's Law gives more cores, not faster cores. To make programs faster, the researcher has to parallelise.

*Concepts: [Moore's Law and Dennard Scaling](../concepts/Moores_Law_and_Dennard_Scaling.md), [HPC Programming Languages](../concepts/HPC_Programming_Languages.md)*

## Question 2 (30 marks)

### 2(a) The following C program calculates the mean and variance of an array of values. It also finds the maximum value in the array. Show how to parallelise the `for` loop using OpenMP. *(10 marks)*

```c
#include <stdio.h>
#define N 5

int main(){
    float x[N]={1.0, 4.0, 6.0, 9.0, 3.0};
    float sum, sum2;
    float mean, sigma2, max_val;
    int i;

    sum     =0.0;
    sum2    =0.0;
    max_val =-1.0e37;

    for (i=0; i<N; i++){
        sum  +=x[i];
        sum2 +=(x[i] * x[i]);
        if (x[i] > max_val) {max_val = x[i];}
    }

    mean   = sum / ( (float) N);
    sigma2 = (sum2 / ( (float) N) ) - (mean * mean);

    printf("Mean = %g\n",   mean);
    printf("Sigma^2= %g\n", sigma2);
    printf("Max= %g\n",     max_val);

    return 0;
}
```

**Answer:**

The serial loop computes three reductions: `sum`, `sum2`, and `max_val`. All three need a `reduction` clause; `max_val` uses the `max` reduction operator (OpenMP 3.1+).

```c
#include <stdio.h>
#include <omp.h>

#define N 5

int main(){
    float x[N] = {1.0, 4.0, 6.0, 9.0, 3.0};
    float sum, sum2;
    float mean, sigma2, max_val;
    int i;

    sum     = 0.0;
    sum2    = 0.0;
    max_val = -1.0e37;

    #pragma omp parallel for default(none) shared(x) \
                             reduction(+:sum, sum2) reduction(max:max_val)
    for (i = 0; i < N; i++){
        sum  += x[i];
        sum2 += (x[i] * x[i]);
        if (x[i] > max_val) { max_val = x[i]; }
    }

    mean   = sum / ((float) N);
    sigma2 = (sum2 / ((float) N)) - (mean * mean);

    printf("Mean = %g\n",   mean);
    printf("Sigma^2= %g\n", sigma2);
    printf("Max= %g\n",     max_val);

    return 0;
}
```

**Justification:**
- `default(none)` forces explicit scoping, catching accidental data races.
- `x` is read-only and `shared`.
- `i` is the loop counter and is implicitly `private`.
- `sum` and `sum2` are floating-point sums → `reduction(+ : sum, sum2)` gives each thread a private partial sum and combines them after the loop.
- `max_val` is updated by `if (x[i] > max_val)` — a classic data race if shared. The `reduction(max : max_val)` clause gives each thread a private maximum and combines them at the end.

*(Note: with only `N = 5` iterations the parallel version is slower than serial in practice; the question asks for the technique, not for a meaningful speed-up.)*

*Concepts: [Parallel Loops in OpenMP](../concepts/Parallel_Loops_OpenMP.md), [Variable Scoping in OpenMP](../concepts/Variable_Scoping_OpenMP.md), [Data Dependencies and Data Races](../concepts/Data_Dependencies.md)*

### 2(b) Compare the OpenMP and MPI methods for writing parallel programs. Discuss the strengths and weaknesses of each method. *(10 marks)*

**Answer:**

| Feature | OpenMP | MPI |
| :--- | :--- | :--- |
| **Memory model** | Shared memory | Distributed memory |
| **How parallelism is expressed** | Compiler directives (`#pragma omp ...`) on otherwise-serial code | Library calls (`MPI_Send`, `MPI_Recv`, `MPI_Bcast`, …) |
| **Where it can run** | Within a single shared-memory node only | Across an entire cluster, also fine within a node |
| **Programmer effort** | Low — incremental, often just one or two pragmas | High — must explicitly distribute data, code halo exchanges, etc. |
| **Communication** | Implicit (shared variables) | Explicit (messages, payload, source/dest, tag) |
| **Scaling ceiling** | Bound by single-node core count and memory | Scales to thousands of nodes |

**Strengths of OpenMP:**
- Easy to add to an existing serial code.
- No marshalling/unmarshalling of data — threads see all variables.
- Excellent for fine-grained loop parallelism.
- Fast, low-overhead synchronisation through shared memory.

**Weaknesses of OpenMP:**
- Cannot scale beyond a single shared-memory node.
- Subtle bugs from data races and incorrect variable scoping.

**Strengths of MPI:**
- Scales across distributed-memory clusters — the only path to thousands or millions of cores.
- Forces clean data distribution, which often produces better-engineered code.
- Portable across virtually every HPC architecture.

**Weaknesses of MPI:**
- High programming effort: domain decomposition, halo exchange, collective design.
- Easy to write deadlocks (e.g. mismatched `Send`/`Recv`, partial collectives).
- Each process duplicates code data and halos, increasing total memory footprint.

**Hybrid MPI+OpenMP** combines both: MPI between nodes, OpenMP within a node. It cuts memory overhead and reduces the number of MPI messages on the network.

*Concepts: [OpenMP vs. MPI](../comparisons/OpenMP_vs_MPI.md), [Hybrid Parallelism (MPI + OpenMP)](../concepts/Hybrid_Parallelism_MPI_OpenMP.md)*

### 2(c) Show how the following sparse matrix would be represented in compressed sparse row format with rows and columns numbered starting from one. *(10 marks)*

$$
\begin{pmatrix}
6 & 0 & 7 & 0 \\
0 & 2 & 0 & 0 \\
3 & 0 & 0 & 0 \\
0 & 0 & 0 & 1
\end{pmatrix}
$$

**Answer:**

Walking through the matrix row-by-row, listing only non-zero entries (rows and columns numbered from 1):

| Row | Non-zeros |
| :--: | :--- |
| 1 | (col 1, val 6), (col 3, val 7) |
| 2 | (col 2, val 2) |
| 3 | (col 1, val 3) |
| 4 | (col 4, val 1) |

`nnz = 5`.

```
vals    = [ 6,  7,  2,  3,  1 ]
col_ind = [ 1,  3,  2,  1,  4 ]
row_ptr = [ 1,  3,  4,  5,  6 ]
```

**`row_ptr` derivation:**
- Row 1 starts at `vals[1]` (1-indexed) → `row_ptr[1] = 1`.
- Row 1 has 2 entries, so row 2 starts at `1 + 2 = 3` → `row_ptr[2] = 3`.
- Row 2 has 1 entry, so row 3 starts at `3 + 1 = 4` → `row_ptr[3] = 4`.
- Row 3 has 1 entry, so row 4 starts at `4 + 1 = 5` → `row_ptr[4] = 5`.
- Sentinel: `row_ptr[5] = nnz + 1 = 6`.

*Concepts: [Sparse Matrices and CSR](../concepts/Sparse_Matrices_and_CSR.md)*

## Question 3 (30 marks)

### 3(a) You have been given the task of parallelising a program and need to achieve a strong scaling speed-up of at least 100.

#### (i) What is the smallest parallel fraction which permits a speed-up of 100? *(3 marks)*

**Answer:**

By Amdahl's Law, the maximum speed-up (as $N \to \infty$) is $\frac{1}{s}$ where $s = 1 - p$. We need:
$$\frac{1}{1-p} \ge 100 \quad \Rightarrow \quad 1 - p \le 0.01 \quad \Rightarrow \quad p \ge 0.99$$

The **smallest parallel fraction is $p = 0.99$** (equivalently, the serial fraction $s$ must be at most 0.01).

*Concepts: [Parallel Scaling](../concepts/Parallel_Scaling.md)*

---

#### (ii) You parallelise the program and achieve a parallel fraction of p = 0.999. How many processors are required to obtain a speed up of 100 assuming idealised strong scaling? *(3 marks)*

**Answer:**

Amdahl's Law: $S_N = \dfrac{1}{s + p/N}$ with $s = 0.001$.

$$100 = \frac{1}{0.001 + 0.999/N}$$
$$0.001 + \frac{0.999}{N} = 0.01$$
$$\frac{0.999}{N} = 0.009$$
$$N = \frac{0.999}{0.009} = 111$$

**N = 111 processors.**

---

#### (iii) If the total parallel overhead is equivalent to 0.5% of the serial run time, what is the smallest number of processors required to achieve a speed up of at least 100? *(4 marks)*

**Answer:**

With overheads, $S_N = \dfrac{1}{s + p/N + V/T_0}$ where $V/T_0 = 0.005$.

$$100 = \frac{1}{0.001 + 0.999/N + 0.005}$$
$$0.001 + \frac{0.999}{N} + 0.005 = 0.01$$
$$\frac{0.999}{N} = 0.004$$
$$N = \frac{0.999}{0.004} = 249.75$$

Since $N$ must be a positive integer and we need speed-up *at least* 100, round up: **N = 250 processors**.

*Concepts: [Parallel Scaling](../concepts/Parallel_Scaling.md)*

### 3(b) You are given the task of planning an upgrade to an HPC cluster which involves replacing all the compute nodes. The total number of compute nodes will not change and there will be no other changes to the hardware. You need to choose between two compute node specifications:

- **Compute node type 1:** number of sockets=2, number of cores per socket=64, processor clock frequency=2.2 GHz, floating point operations per cycle=4
- **Compute node type 2:** number of sockets=2, number of cores per socket=28, processor clock frequency=2.8 GHz, floating point operations per cycle=16

The cost of the compute nodes is the same so you need to make a choice based on performance.

#### (i) Which type of compute node has the higher theoretical peak performance? Show your working. *(5 marks)*

**Answer:**

Using $R_{peak} = N_{sockets} \cdot N_{cores} \cdot R_{clock} \cdot N_{ops/cycle}$:

**Type 1:**
$$R_{peak} = 2 \times 64 \times 2.2 \times 10^9 \times 4 = 1.1264 \times 10^{12}\ \text{flop/s} = 1126.4\ \text{GFlop/s}$$

**Type 2:**
$$R_{peak} = 2 \times 28 \times 2.8 \times 10^9 \times 16 = 2.5088 \times 10^{12}\ \text{flop/s} = 2508.8\ \text{GFlop/s}$$

**Type 2 has the higher theoretical peak performance** (≈ 2.23× Type 1's peak).

*Concepts: [Performance Metrics and Top500](../concepts/Performance_Metrics_and_Top500.md)*

---

#### (ii) You know that the application which will run on the system has not been optimised to make effective use of the instruction set in modern processors. This means that the application is not able to use vector instructions and fused-multiply-add instructions. Explain whether this would influence your choice of compute node and why. *(3 marks)*

**Answer:**

**Yes, this changes the choice.** Type 2's high `ops/cycle = 16` count (= 256-bit/512-bit AVX with FMA) is what gives it the lead. If the application can only issue scalar, non-FMA instructions, it cannot exploit this — its effective `ops/cycle` collapses to roughly 1–2 per core regardless of hardware.

Effective comparison without vector/FMA (approximate, using ~1 op/cycle):
- Type 1 effective: $2 \times 64 \times 2.2\ \text{GHz} \approx 282\ \text{GFlop/s}$
- Type 2 effective: $2 \times 28 \times 2.8\ \text{GHz} \approx 157\ \text{GFlop/s}$

Now **Type 1 wins** because it has more cores at a similar clock rate. Choose Type 1 for an unoptimised, scalar code.

*Concepts: [Floating Point Arithmetic](../concepts/Floating_Point_Arithmetic.md), [Performance Metrics and Top500](../concepts/Performance_Metrics_and_Top500.md)*

---

#### (iii) You know that the application which will run on the system makes extensive use of sparse matrix operations. What extra piece of information would you need to know in order to assess the suitability of the compute nodes for running this application and why is this information important? *(2 marks)*

**Answer:**

Sparse matrix operations are typically **memory-bound** (low arithmetic intensity, indirect addressing). Theoretical peak FLOPs are largely irrelevant. The missing information is **memory bandwidth** (and ideally cache size/structure) of each compute node — the application's performance will be set by how fast each node can stream data from DRAM, not by FLOP rate.

*Concepts: [Arithmetic Intensity and Roofline Model](../concepts/Arithmetic_Intensity_and_Roofline_Model.md), [Sparse Matrices and CSR](../concepts/Sparse_Matrices_and_CSR.md)*

### 3(c) Consider a two-dimensional partial differential equation (PDE) solver which solves a PDE with one unknown variable u. To solve the PDE it is necessary to calculate values for the second derivative:

$$\nabla^2u = \frac{\partial^2 u}{\partial x^2} + \frac{\partial^2 u}{\partial y^2}$$

This expression is evaluated using the finite difference stencil:

```
del2u = (u(i+1,j) - 2*u(i,j) + u(i-1,j))/(dx*dx) +
        (u(i,j+1) - 2*u(i,j) + u(i,j-1))/(dy*dy)
```

The computation domain for the whole calculation comprises 8192 × 8192 grid points. In addition to these grid points each sub-domain stores halo values which are copies of the data on neighbouring processors.

#### (i) The calculation is domain decomposed in an 8 × 8 configuration such that there are 64 square sub-domains of equal size. How many values need to be exchanged when one halo is sent to one neighbouring process? *(2 marks)*

**Answer:**

Each sub-domain is $\frac{8192}{8} \times \frac{8192}{8} = 1024 \times 1024$ grid points. A halo (one edge sent to one neighbour) is one strip of length 1024:

$$\text{Values per halo} = 1024$$

*Concepts: [Domain Decomposition](../concepts/Domain_Decomposition.md), [Domain Decomposition Overheads](../concepts/Domain_Decomposition_Overheads.md)*

---

#### (ii) How many bytes need to be sent if the values are represented using IEEE754 double precision? *(2 marks)*

**Answer:**

Double precision = 8 bytes per value:
$$1024 \times 8 = 8192\ \text{bytes}$$

*Concepts: [Floating Point Arithmetic](../concepts/Floating_Point_Arithmetic.md)*

---

#### (iii) What is the transmission time for sending a single halo over a network with a latency of 1µs (10⁻⁶s) and a bandwidth of 10.5 GB/s (10.5 gigabytes per second)? Show your working. *(3 marks)*

**Answer:**

Using $t = L + M/B$ with $M = 8192$ bytes, $B = 10.5 \times 10^9$ B/s:

$$t = 1.0 \times 10^{-6} + \frac{8192}{10.5 \times 10^9}$$
$$t = 1.0 \times 10^{-6} + 7.802 \times 10^{-7}$$
$$\boxed{t \approx 1.78 \times 10^{-6}\ \text{s} = 1.78\ \mu\text{s}}$$

*Concepts: [Interconnects and Network Topologies](../concepts/Interconnects_and_Network_Topologies.md)*

---

#### (iv) The domain decomposition is modified so that the domain is decomposed in a 16 × 16 configuration such that there are 256 square sub-domains of equal size. What is the transmission time for one halo with the modified domain decomposition? *(3 marks)*

**Answer:**

Sub-domain becomes $\frac{8192}{16} \times \frac{8192}{16} = 512 \times 512$. Halo length = 512.
- Bytes: $512 \times 8 = 4096$.

$$t = 1.0 \times 10^{-6} + \frac{4096}{10.5 \times 10^9} = 1.0 \times 10^{-6} + 3.901 \times 10^{-7}$$
$$\boxed{t \approx 1.39 \times 10^{-6}\ \text{s} = 1.39\ \mu\text{s}}$$

**Observation:** halving the halo only reduces time from 1.78 µs to 1.39 µs because **latency dominates** at this message size — the bandwidth term is small relative to $L$. Meanwhile the *computation* per sub-domain has fallen by a factor of 4, so the **communication-to-computation ratio gets worse** as we strong-scale further.

*Concepts: [Interconnects and Network Topologies](../concepts/Interconnects_and_Network_Topologies.md), [Domain Decomposition Overheads](../concepts/Domain_Decomposition_Overheads.md)*
