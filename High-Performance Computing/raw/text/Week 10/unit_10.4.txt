Programming a GPU for HPC
35

--- page 3 ---
• The CPU and GPU address separate memory
• GPU memory smaller than host memory
• Need to transfer data between CPU memory and GPU memory
• Transfer is over the PCIe bus is slow: need to minimise transfers
• Small tasks may be faster on CPU: there is a minimum granularity for offloading to be 
beneficial
• Need to consider data locality to get good performance
Data movement
36

--- page 4 ---
• CPUs hide memory latency using cache
• GPUs over-commit cores to hide memory latency
• Context switching is much faster on a GPU than a CPU (over-committing cores is not 
beneficial to performance on a CPU)
• Require a higher degree of parallelism for GPUs
Hiding memory latency
37

--- page 5 ---
• GPUs are suited to data parallel workloads where each thread carries out the same 
operation on different data items (Single Instruction Multiple Data)
• Branching is bad for performance 
- Branch divergence within a warp causes performance degradation as threads need to execute 
in lock step
- GPUs do not have the branch prediction available in CPUs 
Avoiding branching
38

--- page 6 ---
• Traditional programming languages alone are not able to make use of GPUs so other 
programming models have been developed: 
- OpenMP: compiler directives (C/C++, Fortran)
- OpenCL: C based
- OpenACC: compiler directives (C/C++, Fortran)
- CUDA: C/C++, Fortran based (from NVIDIA)
- HIP: C++ based (from AMD)
- Data Parallel C++ (from Intel)
