--- page 6 ---
• Cache helps hide memory latency provided we make efficient use of the data it holds
• Cache relies on the predictability provided by locality of reference: 
- Data nearby in memory is more likely to be used (spatial locality)
- Data will be re-used within a relatively short time window (temporal locality) 
Temporal and spatial locality  
14

--- page 7 ---
• Data is transferred from main memory to cache in fixed size blocks called cache lines
• Cache lines on Intel x86_64 processors are 64 bytes (512 bits)
• When we access adjacent memory locations the next piece of data required is likely to 
be in cache already (a cache hit)
• When the next piece of data is not in cache it needs to be loaded from main memory 
(cache miss)
• Read from memory locations sequentially to use the whole cache line (access arrays 
stride one)

--- page 11 ---
• We can split the work up into cache-sized blocks and work on one block before 
moving on to the next one
• All the data required can be held in cache so can be accessed much faster
• This exploits temporal locality: re-use data in the block within a short time window
• See https://www.intel.com/content/www/us/en/developer/articles/technical/loop-
optimizations-where-blocks-are-required.html for more on cache blocking

--- page 12 ---
• When debugging a program we work with a small test case which runs in a 
manageable time
• Reducing the problem size can change the performance characteristics in a non-
uniform manner (e.g. if a significant data structure fits into cache)
• When testing performance we need a test case with a representative memory 
footprint
• For example when testing the performance of a PDE solver use a representative 
domain size but run for fewer time steps

--- page 13 ---
• Compilers can optimise a program by making changes to the source code to produce 
equivalent results faster
• These optimisations can include optimisation of memory use e.g. loop interchange 
and cache blocking
• The optimisation level can be controlled with a command line flag (-O3 turns on all 
optimisations in gcc)
• See https://gcc.gnu.org/onlinedocs/gcc/Optimize-Options.html for more on GCC 
optimisations 
Compiler optimisations  
21

--- page 14 ---
• Accessing main memory takes ~100 CPU cycles but this latency can be hidden by the 
memory cache
• Achieving good performance on modern processors requires exploiting locality of 
reference to make efficient use of cache:
- Spatial locality: order of loop nesting matters
- Temporal locality: cache blocking
• Test cases for performance analysis should use a realistic problem size as 
performance can scale non-uniformly with memory footprint
• See SAB section 6.5 “Memory Hierarchy”
