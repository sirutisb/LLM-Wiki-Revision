Parallelising loops with OpenMP
22

--- page 23 ---
• shared: all threads access the same copy of the variable
• private: each thread has its own copy of the variable which is uninitialised on 
entry to the parallel region
• firstprivate: each thread has its own copy of the variable which is initialised 
with the value from outside the parallel region
• lastprivate: each thread has its own copy of the variable and the variable is 
updated at the end of the parallel region with the value from the last loop 
iteration
• reduction: each thread has a private copy of the variable and the private 
copies are combined with a reduction operator at the end of the parallel region

- Output dependence (write-after-write): corrected by using a lastprivate

--- page 36 ---
• A number of factors can contribute to real-world parallel performance being 
less than the idealised peak performance:
- Starvation: insufficient parallel work to keep the processors busy (insufficient parallel 
work, uneven distribution of work) 
- Latency: time taken for information to travel from one part of the system to another 
(e.g. accessing memory, sending messages)
- Overhead: work required in addition to the computation (e.g. starting and stopping 
OpenMP parallel regions)
- Waiting: multiple threads/processes accessing a shared resource (e.g. contention for 
memory or network bandwidth)

• Parallel programs contain points synchronisation points (e.g. blocking 
collectives, barriers at the end of OpenMP parallel regions)
• If the amount of work per thread/process is not approximately equal then 
some threads/processes will have to wait at the synchronisation point
• This results in inefficient use of resources because some processors are idle
• With an MPI program the manager-worker pattern can be used to load 
balance at run time (unit 8.2 and workshop 8)
