--- page 2 ---
• MPI_Send and MPI_Recv are blocking communications
• MPI_Send will block until the message is received by the destination process or until it 
is safe to change the send buffer
• MPI_Recv will block until the message is received This behaviour helps write correct 
programs (avoid overwriting a send buffer before it is safe to do so)
• If send and receive operations are not correctly matched there may be a deadlock

--- page 5 ---
• MPI_Send MPI_Isend and MPI_Irecv are non-blocking versions of MPI_Send 
and MPI_Recv
• Non-blocking communication can avoid deadlocks as we don’t have to match sends 
and receives so carefully
• It can also be more efficient as we can overlap computation and communication (do 
something else whilst waiting for a message)
• We need to be careful that the program works correctly and only change data in the 
send buffer once it has been received
• Make a call to MPI_Wait/MPI_Waitall at the point in the program when the 
communications need to have completed

• The collectives in unit 4.3 are blocking calls 
• All processes in the specified communicator must reach the collective communication 
or there will be a deadlock
• A blocking collective synchronises all the processes in the communicator
 
• Non-blocking collectives were introduced in version 3 of the MPI standard

• This can cause a deadlock if sends and receives don’t match
