Performance degradation
8
• A number of factors can contribute to real-world parallel performance being less than 
the idealised peak performance: 
- Starvation: insufficient parallel work to keep the processors busy (insufficient parallel work, 
uneven distribution of work)
- Latency: time taken for information to travel from one part of the system to another (e.g. 
accessing memory, sending messages)
- Overhead: work required in addition to the computation (e.g. starting and stopping OpenMP 
parallel regions)
- Waiting: multiple threads/processes accessing a shared resource (e.g. contention for memory 
or network bandwidth)

Strong scaling
12
• Extra performance lets us do the same calculation quicker
• In strong scaling we keep the total problem size fixed and increase number of 
processors to reduce the run time
• For example parallelising a 1024x1024 domain using domain decomposition: 
- 4 processors: 2x2 sub-domains with 512x512 points
- 16 processors: 4x4 sub-domains with 256x256 points
- 64 processors: 8x8 sub-domains with 128x128 points

Amdahl’s law
13
• How much faster can we calculate a solution to a given problem by parallelising the 
calculation?
• The parallel speed-up is limited by the fraction of the program which can be 
parallelised (Amdahl, 1967)
• We can use this to work out the speed-up under the assumption that the only limiting 
factor is the parallel fraction
